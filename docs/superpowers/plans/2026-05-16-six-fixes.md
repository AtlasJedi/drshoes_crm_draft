## Root cause analysis (per fix)

### Fix A — Pozycje sum ≠ Wycena, no row by default
- `apps/web/app/(admin)/admin/orders/new/_components/NewOrderForm.tsx`
  - `items` initialized to `[]`; user must click "+ Dodaj pozycję".
  - `quotedPricePln` is its own input bound to `setQuotedPricePln`, decoupled from items.
- Backend: `OrderController.create` accepts `quotedPriceCents` and `items[]` independently; happy with either source.

### Fix B — No RODO toggle in ClientCreateModal
- `apps/web/components/clients/ClientCreateModal.tsx` form only collects firstName / lastName / phone / email. `CreateClientRequest` (in `apps/web/lib/clients/types.ts`) has no `rodoConsent` field.
- Backend `Client.rodoConsentAt` is `Instant` (nullable); `ClientService.create` does not stamp it.

### Fix C — Triggers don't fire / no real email path
- Email provider defaults to **NOOP** (`application.yaml`: `drshoes.email.provider: NOOP`).
- `EmailGatewayProperties.Provider` already declares `SMTP`, but **no `SmtpEmailGateway` implementation exists** — only `LoggingEmailGateway` and `PostmarkEmailGateway`.
- `spring-boot-starter-mail` not in any pom.
- `V006__seed_templates_triggers_and_trigger_fire.sql` seeds STATUS_CHANGE triggers for `PRZYJETE` and `GOTOWE_DO_ODBIORU` only — **no ORDER_CREATED trigger**. Owner wants "create order → email fires" so we need a new trigger row plus an event type and a `TriggerEngine.onOrderCreated` hook.
  - `TriggerEvent` enum currently: `STATUS_CHANGE`, `BEFORE_PICKUP_X_DAYS`, `AFTER_HANDOVER_Y_DAYS`. Add `ORDER_CREATED`.
- The user also wants the existing triggers to actually deliver. Once SMTP is wired and `drshoes.email.provider=SMTP`, the existing STATUS_CHANGE triggers will deliver real email too.

### Fix D — Note not visible in timeline
- `TimelineEventCurator.curate` (line 168): `if ("POST".equals(method) && status == 200 && path.endsWith("/notes"))`.
- `OrderNotesController.add` returns `ResponseEntity.created(...)` → **HTTP 201**. The `status == 200` guard rejects every real note row. The note IS persisted in `audit_log` but never classified as `ORDER_NOTE`, so the OrderDrawerNotes filter (`ev.kind === "ORDER_NOTE"`) skips it.
- Fix: accept `status == 200 || status == 201`. Optionally also flip the controller to 200, but accepting 201 in the curator is the safer, minimally-invasive fix.

### Fix E — Note composer auto-fires / always-visible
- Composer is always rendered at the bottom of the drawer. Owner wants it hidden behind a "Dodaj wpis do historii" button. The current section header is a `<div>`, not a button. The submit ("dodaj wpis") button already exists at line 90 — so this is purely a collapsibility change.

### Fix F — Status change actions inverted
- `OrderDrawerStatusChanger.tsx` line 183:
  ```tsx
  onConfirm={(_sendTriggers, note) => { void handleConfirm(note); }}
  ```
  The `sendTriggers` flag from the dialog is **discarded**.
- `handleConfirm` calls `changeStatus(order.id, target, order.version, note)` without `sendTriggers`.
- `apps/web/lib/orders/api.ts changeStatus` accepts no `sendTriggers` parameter.
- `ChangeStatusRequest.java` accepts `Boolean sendTriggers` and defaults `null → TRUE` in the compact ctor. Result: every status change fires the trigger, no matter which button you click.
- The user's observation that "Wyślij wiadomość doesn't send" is consistent with: email provider is NOOP, so when they clicked "wyślij wiadomość" they saw no Gmail land — but a MESSAGE_SENT audit row WAS created. The "Tylko zmień status" complaint is the unambiguous bug.

## Risks / unknowns

- **SMTP Gmail rate limits**: 500/day for non-Workspace accounts. Fine for testing.
- **TimelineEventCurator** may have other call sites filtering on `status == 200` that should accept 201 too — quick `grep` during slice D to confirm only the `/notes` path needs the bump.
- **V-number collision**: list `db/migration` before authoring V019. If a newer one exists, bump.
- **`spring.mail.host` empty**: if SMTP_USER unset, the JavaMailSender bean still gets created and will throw on first send. Slice C must guard: if `provider=SMTP` but no host/user configured, log a clear WARN at startup and fall back to LoggingEmailGateway. This keeps tests / local dev clean.
- **Existing seed templates**: M2 added templates with `name = 'Zlecenie przyjete (EMAIL)'` etc. Verify the schema of `message_template` (channel column? body column?) before authoring V019 INSERT.
