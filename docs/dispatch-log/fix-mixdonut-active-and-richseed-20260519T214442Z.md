## 1. SQL diff — `countByItemKind` (OrderRepository.java)

**Before:**
```sql
SELECT oi.kind AS kind, COUNT(*) AS cnt
FROM order_item oi
JOIN order_ o ON o.id = oi.order_id
WHERE o.deleted_at IS NULL
GROUP BY oi.kind
```

**After:**
```sql
SELECT oi.kind AS kind, COUNT(*) AS cnt
FROM order_item oi
JOIN order_ o ON o.id = oi.order_id
WHERE o.deleted_at IS NULL
  AND o.status NOT IN ('WYDANE','ANULOWANE','GOTOWE_DO_ODBIORU')
GROUP BY oi.kind
```

Semantic change: Mix donut now shows "work still to do" — items belonging to
actively-in-progress orders only. Terminal/ready statuses excluded.

## 3. Wipe script

**Path:** `backend/app/src/main/resources/db/cleanup/wipe_demo_data.sql`

**Tables truncated (in FK dependency order):**
1. `message`
2. `message_thread`
3. `photo`
4. `order_item`
5. `order_`
6. `audit_log`
7. `client_phone`
8. `client`
9. `order_code_counter` — **also reset** so next seed starts at DR-YYYY-0001

**Not touched:** `user_`, `setting`, `template`, `trigger`, Flyway tables.

**FK / sequence gotcha:** Order codes are generated by `next_order_code(p_year)` which
reads/writes `order_code_counter`. TRUNCATE on `order_` does NOT reset this counter.
The wipe script explicitly truncates `order_code_counter` so the reseed starts from
`DR-2026-0001` rather than continuing from wherever the previous run left off.
