## Decisions

1. **`setSubmitting(true)` placement in adhoc path:** In ad-hoc mode `setSubmitting` is called before `createClient` (at the start of the async block). In existing-client mode it's called just before `createOrder`. This avoids a double-set and keeps the existing path logically intact. A small structural difference from the original code — acceptable since both paths result in correct `submitting` state management.

2. **`setSubmitting(true)` redundant guard in existing-client path:** After the ad-hoc block resolves `resolvedClientId`, the existing-client path hits `setSubmitting(true)` again. This is a no-op for the existing-client path (it was already set) but harmless. Could be extracted to a single `setSubmitting(true)` before the branch — deferred as minor cleanup.

3. **Name splitting:** `adhocName.trim().split(/\s+/)` — first token → `firstName`, rest joined → `lastName`. Single word → `lastName = null`. Matches plan spec exactly.

4. **`rodoConsent: true` hardcoded** in `createClient` call — matches plan spec and `ClientCreateModal` precedent.

5. **No `<NewClientFields>` sub-component extracted** — inline JSX per anti-bloat constraint. Net new LOC in implementation: ~90 (slightly over the 80 LOC target due to the mode-switcher button block needing full Tailwind conditional classes). No logic was added beyond the plan spec.

6. **Pre-existing test failures** (useKanbanDnd + OrderDrawer): confirmed pre-existing on HEAD before this task via `git stash` baseline. Not introduced by this change.
