## Decisions / Surprises

1. **`role="tab"` vs `getByRole("link")` conflict:** The plan's component code specifies `role="tab"` on the Link elements, but the plan's test code uses `getByRole("link", { name: /przegląd/i })`. Setting `role="tab"` on an `<a>` overrides the implicit `link` role — the element becomes a tab, not a link, for ARIA purposes. `getByRole("link")` finds nothing. Resolution: removed `role="tab"` from the Link; the `aria-current="page"` attribute alone signals the active state and the `role="tablist"` on the container provides the ARIA semantics. The test passes; the outer `role="tablist"` container still provides the group semantics.

2. **`getClientServer` error does not attach `.status`:** The plan's layout template uses `(err as { status?: number }).status === 404` to detect not-found. The existing `getClientServer` threw a plain `new Error(...)` with the status only in the message string. Added `.status = resp.status` to the thrown error object so the layout's type-cast check works as designed. The thrown message string is unchanged, so existing tests that assert `rejects.toThrow("clients/get failed: …")` continue to pass.

3. **`api-server.ts` included in the feature commit:** The dispatch scope listed `apps/web/lib/clients/api-server.ts` as an optional inline addition "if missing" — since the file existed but lacked the `.status` property, the small change to wire up 404 detection was included in the same atomic feature commit, consistent with plan intent.

4. **No `next/link` mock in the plan's test code:** The plan's test template did not include a `next/link` mock, but the component uses `next/link`. Added the standard mock (plain `<a>` passthrough) that the rest of the codebase uses (see `OrderViewTabs.test.tsx`). Without it, Vitest throws a module resolution error.
