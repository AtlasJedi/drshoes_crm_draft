## Decisions / deviations from plan

### 1. WebhookEvent extended with errorCode + errorMessage (V011 migration)

**Plan said:** Task 4-7 creates PostmarkWebhookController; WebhookEvent still has 6 fields.
The plan's `fromPostmark` mapper passes 8 arguments but the actual record had 6.

**Decision:** Added errorCode + errorMessage to WebhookEvent (making it 8 fields) as part
of this task. Created V011 migration to add matching columns to webhook_event table.
The 4-2 dispatch log confirmed only 6 fields were added in that task. This is a forward pull
— necessary for the mapper and controller to compile.

**WebhookEvent.status made nullable:** The plan showed `DROPPED` events as having `null` status
(non-delivery record types like Click). The existing compact assertion `requireNonNull(status)`
had to be relaxed. Updated WebhookEventTest: replaced `statusRequired` with `statusNullable`,
added `rawPayloadRequired` test.

### 2. WebhookStatusReconciler created as part of 4-7 (not 4-9)

**Plan said:** Task 4-9 (TWO-STAGE) creates WebhookStatusReconciler, AppliedOutcome, ReconcileResult.
Task 4-7's controller calls `reconciler.apply(event)` — so 4-9's output is a 4-7 compile dependency.

**Decision:** Created the reconciler in 4-7 to make the controller compile and the IT pass.
4-9 is now effectively pre-shipped. The reconciler at 186 LOC subsumes all 4-9 logic
(state-guarded UPDATE, DROPPED/DEDUP/NO_MESSAGE/NO_TRANSITION/APPLIED outcomes, audit row).

**Implication for 4-9:** Task 4-9 (TWO-STAGE review) should be a quality-review-only pass —
no new code needed. The reconciler is already green and tested through the controller IT.

### 3. Audit row written via AuditLogWriter, not @Audited

**Plan said:** `@Audited(parent = ...)` on reconciler to wire timeline events.

**Problem:** SpEL expressions in `@Audited` only see method args and return value.
The `apply(WebhookEvent event)` method's return is `ReconcileResult` which carries `messageId`
(not `orderId`). Getting `orderId` from `messageId` would require a second DB lookup via SpEL
context — not supported by SimpleEvaluationContext.

**Decision:** Injected `AuditLogWriter` into `WebhookStatusReconciler`. On APPLIED outcome,
manually calls `auditWriter.write("INTERNAL", "WebhookStatusReconciler#apply", 0, null, null, orderId, null)`.
This places `orderId` as `parent_entity_id` in the audit row, which `AuditLogRepository.findOrderTimelineRows`
picks up via the `parent_entity_id = :orderId` OR clause. The curator matches the synthetic path
pattern `^WebhookStatusReconciler#apply$` and emits MESSAGE_DELIVERED.

AuditLogWriter uses `Propagation.REQUIRES_NEW` so the audit row commits even if the main
transaction rolls back — consistent with how HTTP and @Audited rows are persisted.
