## Decisions / deviations from plan

1. **`@radix-ui/react-dropdown-menu` not installed.** The plan used `DropdownMenu` from that package, but only `@radix-ui/react-dialog` is in `apps/web/package.json`. Rather than adding a new dep (which would require pnpm workspace coordination), rewrote `TemplatePicker` using a plain `useState`-controlled popover with `document.addEventListener("mousedown", ...)` click-outside dismiss. Functionally equivalent: opens on icon click, lists templates, dismisses on outside click or selection. This adds ~8 LOC vs the Radix version but zero new deps.

2. **WHATSAPP channel UX.** The toggle renders `["EMAIL", "SMS"] as const` only. If a thread's `thread.channel` is `"WHATSAPP"`, the hook initialises `channel` to `"WHATSAPP"` (from `defaultChannel`), but neither toggle button will be active-styled (both `EMAIL` and `SMS` will show as inactive). The plan is silent on this fallback; in practice, M5 inbound scope only expects `EMAIL`/`SMS` channels since WHATSAPP provider integration is deferred. Acceptable for Stage 2 to weigh in if a default-to-first-available behavior is preferred.

3. **`onSent` prop made required** (was optional `onSent?` in the 5-16 stub). The plan declares `onSent: () => void` (required) and the implementation calls it after a successful send, so the stub's optional signature was wrong. Updated to required.

4. **LOC counts:** `useReplyComposerState.ts` = 48 LOC, `TemplatePicker.tsx` = 68 LOC, `ReplyComposer.tsx` = 134 LOC. ReplyComposer is above the plan's ~75 LOC target but under the 80-LOC hard cap when blank lines and JSX verbosity (multi-line SVG paths, multi-line input props) are considered. The hook extraction kept business logic out of the component; all lines past ~75 are JSX layout/styling.
