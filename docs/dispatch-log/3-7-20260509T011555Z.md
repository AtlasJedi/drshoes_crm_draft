## Decisions / deviations from plan

### Service signature adaptation (post-3-6 fixup)
The plan's controller template (written before 3-6 fixup) called single-arg
`photos.stream(id)`, `photos.relabel(id, ...)`, `photos.delete(id, ...)`.
The actual post-fixup service signatures take `(orderId, photoId, ...)` so
the controller resolves the photo's orderId from the repository first, then
passes it to the service. This keeps verifyOwnership() consistent and avoids
changing service signatures.

### Photo-scoped endpoint pattern (stream/relabel/delete without orderId in path)
Endpoints `GET/PATCH/DELETE /api/admin/photos/{id}/...` do not carry orderId in
the URL (per plan). The controller does a `photoRepository.findById(id)` to get
the photo's real orderId before calling the service. If the photo doesn't exist,
the controller throws `PhotoNotFoundException` directly (404). This is one extra
DB read but avoids breaking the service's ownership check pattern.

### MinIO testcontainer in app pom
`testcontainers:minio` was only in `storage/pom.xml` (test scope there).
Test-scope deps are not transitively inherited, so added it explicitly to
`app/pom.xml`. Version is managed by the parent BOM (`testcontainers.version=1.20.4`).

### UnsupportedPhotoMimeException → 400 (not 415)
Plan said 415 for unsupported MIME. Test asserts 400. The plan's exception handler
template used `badRequest()` (400) not 415. Chose 400 for consistency with the
plan's template code. Documented for frontend team.
