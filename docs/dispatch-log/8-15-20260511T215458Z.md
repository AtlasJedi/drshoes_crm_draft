## Decisions

1. **`BatchSpanProcessor` from `sdk-trace-web` not `sdk-trace-base`:** The `^1.28.0` spec for `sdk-trace-base` resolved to `1.28.0` while `sdk-trace-web@^1.28.0` resolved to `1.30.1` (which internally uses `sdk-trace-base@1.30.1`). Importing `BatchSpanProcessor` from the separate `sdk-trace-base@1.28.0` caused a TypeScript private-property type conflict. Fix: import `BatchSpanProcessor` from `sdk-trace-web` (which re-exports it from its own bundled version). Dropped explicit `sdk-trace-base` dep from `package.json`. **New errata** for dispatch log.

2. **`ZoneContextManager` omitted:** Plan errata-aware note listed this as a potential fallback. `context-zone` has `zone.js` as a direct dependency, but Next.js App Router does not guarantee zone.js availability in the client bundle. Using `WebTracerProvider.register()` with no `contextManager` option defaults to `StackContextManager` in the browser, which is safe and simpler.

3. **`BrowserOtelInit` client component wrapper:** The plan discussed a possible alternative to bare `import "@/lib/otel/browser-client"` in the server layout. A `"use client"` directive on a utility module imported from a server component creates a client boundary, but only renders in the client bundle if Next.js detects it as part of a client subtree. Wrapping in a `BrowserOtelInit.tsx` component that renders `null` is the canonical Next.js 16 pattern for triggering client-only side-effectful imports from a server layout. This is an out-of-plan addition; the plan explicitly listed it as the preferred alternative if the bare import approach proved unreliable.

4. **Proxy does not validate `content-type`:** Any POST body is forwarded to Jaeger. Acceptable for demo scope — Jaeger is internal (docker network only). Stage 2 reviewer should confirm this is acceptable for the threat model.

5. **`propagateTraceHeaderCorsUrls: [/.*/]`:** Propagates `traceparent` to ALL outgoing fetch/XHR requests from the browser, including external third-party domains. Acceptable for demo scope. Stage 2 reviewer: confirm or restrict to same-origin only (e.g. `[/localhost/]` or `[window.location.hostname]`).

1. **No credential leakage:** The proxy does NOT forward `Cookie`, `Authorization`, or other sensitive browser headers to Jaeger — only `content-type` is passed. Confirm this is correct.
2. **CORS:** The route handler adds no CORS headers (`Access-Control-Allow-Origin`, etc.). Browser requests are same-origin to `/api/otlp`, so CORS is not needed. Confirm this is the correct scope.
3. **`propagateTraceHeaderCorsUrls: [/.*/]`:** Propagates `traceparent` to all outgoing fetches. Consider restricting to same-origin for production hardening.
4. **No request size limit:** `req.arrayBuffer()` reads the full body. Next.js default body size limit is 4MB. OTLP batches are typically small (< 100KB). Acceptable?
5. **No auth on `/api/otlp`:** Anyone who can reach the Next.js server can POST arbitrary bytes to Jaeger. In the demo/docker setup Jaeger is not internet-exposed, but confirm acceptable.
6. **`BrowserOtelInit` placement in layout:** Rendered before `AdminSidebar` — confirm no hydration ordering issues.
7. **`"use client"` on `browser-client.ts`:** The directive is present; `BrowserOtelInit.tsx` also has `"use client"`. The import chain is: server layout → `BrowserOtelInit` (client boundary) → `browser-client.ts` (client). This is the correct pattern — confirm.
