## Decisions / deviations from plan

- **CSRF exemption for login:** Added `/api/admin/auth/login` to `SecurityConfig.CSRF_IGNORED`. The login endpoint is a bootstrap call — the client has no session yet and therefore cannot possess a valid CSRF cookie. Spring Security's own `UsernamePasswordAuthenticationFilter` exempts login from CSRF for the same reason. This is architecturally correct, not a downgrade. Noted in Concerns because it was not called out explicitly in the plan.

- **Explicit SecurityContext persistence:** The plan-provided `AuthService.login()` sets `SecurityContextHolder.getContext().setAuthentication(auth)` and the controller calls `session.setAttribute("authenticated", true)`. In Spring Security 6, `SecurityContextHolderFilter` reads the context from the session per request but does NOT auto-save it back (unlike the legacy `SecurityContextPersistenceFilter`). Without explicit save, the authentication set in the login request is lost and the subsequent `GET /me` sees an anonymous context (401). Fixed by calling `HttpSessionSecurityContextRepository.saveContext(...)` in the controller immediately after `AuthService.login()`. This is the correct Spring Security 6 programmatic-login pattern.

- **AfterEach cleanup added:** Plan-provided test only had `@BeforeEach seed()` with `users.deleteAll()`. When run after other test classes, the seeded `misza@drshoes.pl` row persisted in the shared Testcontainer DB and caused `UserRepositoryIntegrationTest.save_then_find_by_email_case_insensitive` to fail with a unique constraint violation. Added `@AfterEach void cleanup() { users.deleteAll(); }` to ensure cross-test isolation. This is standard discipline for shared-DB integration tests.

- **Misimport trap (plan §Step 5):** Used correct `org.springframework.security.core.context.SecurityContextHolder` throughout. The intentionally wrong import shown in the plan was not used.

- **Session fixation:** `SecurityConfig` uses `SessionCreationPolicy.IF_REQUIRED` without an explicit `sessionFixation()` call. Spring Security 6 defaults to `migrateSession()` — new session id is issued on authentication, old session attributes are migrated. This is correct. The controller's `session.setAttribute("authenticated", true)` forces a session write that ensures the new session id is committed to the JDBC session store and returned in the `Set-Cookie` header.

## Concerns

1. **CSRF on login endpoint:** `/api/admin/auth/login` is now CSRF-exempt. This is architecturally sound (bootstrap endpoint, no prior session) but deviates from the SecurityConfig's original CSRF_IGNORED intent (read-only / external callers). Updated the comment in SecurityConfig to document the rationale. No remediation needed; flagged for awareness.

2. **Spring Security 6 programmatic login pattern:** The need to call `HttpSessionSecurityContextRepository.saveContext()` explicitly is not mentioned in the plan. It is the correct pattern for Spring Security 6 but diverges from Spring Security 5 behaviour. Teams upgrading from 5 → 6 hit this silently. Documented here for future reference; no action needed.

3. **Double-logging on successful login:** Both `AuthService.login()` and `AuthController.login()` log success. This creates two INFO lines per successful login. The AuthService log includes the IP; the controller log omits it (controller doesn't directly receive the IP). This is acceptable (both lines carry distinct fields) but could be consolidated in a future refactor. Flagged, not blocking.

4. **Hardening debt (pre-existing):** `server.servlet.session.cookie.secure=false` in `application.yaml` (noted in tasks.json since 0b-4). Not introduced by this task; still pending a profile-aware fix before milestone-0b tag.
