## Decisions / deviations from plan
- Plan test used `loginAsOwner()` from `AdminWebTestBase` then checked `SecurityContextHolder`. That helper only sets a MockMvc `RequestPostProcessor` — it does NOT populate the SecurityContextHolder. Test rewritten to call `AuthService.login()` directly against the real DB; this is the correct approach for verifying the post-login SecurityContext state.
- Plan test expected `owner@drshoes.pl` (seeded via V002); `AdminPrincipalLoginTest` extends `AbstractIntegrationTest` (not `AdminWebTestBase`) and seeds its own user with that email.
- `AuditLogWriter` gained a third overload `write(..., UUID parentEntityId, UUID actorId)` with the two-arg overload delegating to it. Both `persistHttp` and `persistAnnotated` in `AuditLogAspect` now pass `resolveActorId()` so `audit_log.actor_id` is populated for every admin request.
- `AuditLogAspect.resolveActor()` renamed to `resolveActorName()` (now returns String) + new `resolveActorId()` (returns UUID). No behavioural change to log output.
- Three test files (`AdminWebTestBase`, `AuthControllerIntegrationTest`, `RbacIntegrationTest`) updated to call `auditLogs.deleteAll()` before `users.deleteAll()` — necessary because `audit_log.actor_id` now references `user_(id)` via an existing FK constraint (V001 line 287), and actor_id was previously always NULL so the constraint never fired. Now it carries real UUIDs so FK enforcement triggers on user deletion.
- `AuthController.me()` signature changed from `MeResponse` (bare return) to `ResponseEntity<MeResponse>` to handle anonymous/non-AdminPrincipal callers cleanly without throwing. Existing test still passes — `TestRestTemplate` handles `ResponseEntity` transparently.
