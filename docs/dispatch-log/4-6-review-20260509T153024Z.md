## 3. Open Question Answers

### 3a. Can `isNetworkError()` catch `HttpClientErrorException`/`HttpServerErrorException`?

**Answer: No — confirmed safe. Neither exception class can reach `isNetworkError()` via either code path.**

Two layers of protection combine:

1. **`onStatus` suppression.** The lambda `(req, resp) -> { /* suppress */ }` is a non-throwing `ClientHttpResponseErrorHandler` replacement. In Spring Boot 3.4 / Spring 6.x, `RestClient.ResponseSpec.onStatus(predicate, handler)` where the `handler` does NOT throw causes `toEntity()` to complete normally with the raw status and body captured in the `ResponseEntity`. No `HttpClientErrorException` or `HttpServerErrorException` is raised on 4xx/5xx paths — the throw is suppressed before it occurs.

2. **Cause-chain analysis.** `HttpClientErrorException` and `HttpServerErrorException` extend `RestClientResponseException` which extends `RestClientException`. Their cause chain is null (these are constructed with a message string, not by wrapping an `IOException`). Even if they somehow escaped the `onStatus` suppression (they cannot), `isNetworkError()` would return false because:
   - `!(e instanceof ResourceAccessException)` — true, they are not `ResourceAccessException`
   - Cause-chain walk finds no `IOException` — their cause is null
   - `isNetworkError()` returns `false` → `throw e` re-throws; no retry triggered.

The `onStatus` suppression is the primary guard. The `isNetworkError()` cause-chain check is a second line of defense for the `SimpleClientHttpRequestFactory` edge case where `RestClientException` (not `ResourceAccessException`) wraps an `IOException` during header-read. **This design is correct and safe.**

### 3b. Should `SmsApiSmsGateway` (135 LOC) be split?

**Recommendation: Keep as-is. No split warranted.**

`PostmarkEmailGateway` was accepted at 157 LOC in the 4-4 review (C2 DOCUMENTED DEBT) because all helpers shared the same fields and no clean boundary existed. `SmsApiSmsGateway` is leaner at 135 LOC and has exactly the same structure.

If `SmsApiNetworkSender` were extracted (holding `doPost`, `executeWithRetry`, `isNetworkError`), it would be ~85 LOC and require passing `RestClient`, `SmsApiProperties`, and a `Logger` as construction-time dependencies or callbacks — adding complexity without improving readability. The class is already well-organized with clear `// ─── private helpers ───` sections. Meaningful source lines (excluding blank lines and comments) are approximately 85.

The 120 LOC cap is intended to prevent god-classes; 135 LOC for a well-structured class with one clear responsibility (send an SMS via SMSAPI) is documented debt, not a defect.

### 3c. `LinkedHashMap` copy in `send()` — is this clean?

**Recommendation: Keep the copy. It is the cleaner design.**

`SmsApiPayloadMapper.toPayload()` returns a `new LinkedHashMap<>()` — a mutable instance. Technically `send()` could mutate it in-place without the copy. However, keeping the mapper as a pure function (returns a new map, no postcondition on mutability) and doing the copy in the gateway is the better design:

- **Mapper purity is a stronger contract.** Future callers of `SmsApiPayloadMapper.toPayload()` should not need to know that the caller plans to mutate the result. Making the mapper's returned map explicitly mutable would be a leaky implementation detail.
- **The copy is O(n) where n is small** (4–5 fields). Not a performance concern.
- **Dispatcher Decision #2 documented this choice.** It is the right call.

No change needed.

### 3d. Thread-safety of `SmsApiSmsGateway`

**Confirmed: thread-safe.**

1. **`RestClient`** — Spring's `RestClient` is documented as immutable post-build (the builder pattern produces a frozen instance). `RestClient.builder().build()` is safe for concurrent use.
2. **`SmsApiProperties`** — `@ConfigurationProperties` beans are effectively read-only after `ApplicationContext` startup. No setter is called after construction in production. Even if a setter were called, `SmsApiProperties` fields are plain `String`/`int`/`List` — no compound state to corrupt.
3. **`SmsApiSmsGateway` itself** — no mutable fields. `restClient` and `props` are `final`. `log` is a `static final` SLF4J logger (thread-safe). `send()` constructs a fresh `LinkedHashMap` per call.
4. **`isNetworkError()`** — pure static utility, no shared state.

No thread-safety concern.

### Finding #3 — NEEDS_FIX: `SmsApiAutoConfiguration` missing `@AutoConfiguration(before = SmsGatewayAutoConfiguration.class)`

**File:** `backend/libs/sms-gateway/src/main/java/com/drshoes/lib/sms/smsapi/SmsApiAutoConfiguration.java`, line 19

**Severity:** NEEDS_FIX (same structural bug as 4-4 O1 Fixup 2 — latent context-startup failure when `messaging.sms.provider=smsapi`)

**Detail:**

`AutoConfiguration.imports` lists:
```
com.drshoes.lib.sms.SmsGatewayAutoConfiguration       ← line 1 (first = processes first)
com.drshoes.lib.sms.smsapi.SmsApiAutoConfiguration    ← line 2
```

`SmsGatewayAutoConfiguration.loggingSmsGateway` has two conditions:
1. `@ConditionalOnMissingBean(SmsGateway.class)` — passes if no `SmsGateway` bean registered yet
2. `@ConditionalOnProperty(prefix = "drshoes.sms", name = "provider", havingValue = "NOOP", matchIfMissing = true)` — passes when `drshoes.sms.provider` is NOT set (matchIfMissing=true)

When `messaging.sms.provider=smsapi` is set for production:
- `drshoes.sms.provider` is NOT set → condition 2 PASSES
- `SmsGatewayAutoConfiguration` processes FIRST → no `SmsGateway` bean yet → condition 1 PASSES
- `LoggingSmsGateway` is registered as `SmsGateway`
- `SmsApiAutoConfiguration` processes second → registers `SmsApiSmsGateway` as `SmsGateway`
- **Two `SmsGateway` beans → `NoUniqueBeanDefinitionException` at context startup**

This is the identical scenario as 4-4 O1, which was the only required fixup in that review.

**Suggested fix:**
```java
// BEFORE:
@AutoConfiguration
@ConditionalOnProperty(name = "messaging.sms.provider", havingValue = "smsapi")

// AFTER:
@AutoConfiguration(before = com.drshoes.lib.sms.SmsGatewayAutoConfiguration.class)
@ConditionalOnProperty(name = "messaging.sms.provider", havingValue = "smsapi")
```

This ensures `SmsApiSmsGateway` is registered first; `SmsGatewayAutoConfiguration` then sees an existing `SmsGateway` bean via `@ConditionalOnMissingBean` and backs off.
