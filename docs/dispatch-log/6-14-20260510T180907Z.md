## Decisions

1. **Plan/reality mismatch — `CalendarOrderDto.plannedPickupAt: string | null`:**
   The plan's test fixtures used `plannedPickupAt: "..."` as non-null, but the actual type (established by 6-13) declares it `string | null`. The plan's `CalendarMonthGrid` implementation called `localDayFromIso(order.plannedPickupAt)` without a null guard — would crash on unscheduled entries. Fix: added `if (!order.plannedPickupAt) continue;` in the bucket loop. Test fixtures updated to include `receivedAt: null` to satisfy the full type.

2. **`router.push` type cast:** Next.js 16 typed router rejects bare template strings. Used `as Route` cast (pattern established by `OrdersTable.tsx` line 1). `Route` imported from `next`.

3. **`CalendarMonthGrid` at 78 LOC:** Slightly under the 80 LOC cap. Grid math (`monthStartOffset`, `daysInMonth`, `localDayFromIso`) is genuinely irreducible — extraction would add indirection without reducing complexity.

4. **`utils.ts` extracted separately:** Keeps `CalendarCell.tsx` under cap and allows future reuse by other calendar surface components (week/day views in later milestones).

5. **`WSTEPNIE_PRZYJETE` mapped to `--blue`:** Plan was silent on this status in the color palette (admin.jsx:503 only covers 6 statuses). Grouped with `PRZYJETE` as blue since it's a pre-accepted state. Consistent with `STATUS_PILL_CLASS` in `status.ts`.
