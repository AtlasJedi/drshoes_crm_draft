## Decisions / deviations from plan

1. **`MessageTemplateEntity` constructor visibility**: Plan skeleton shows `protected` constructor,
   but `TemplateService` lives in `com.drshoes.app.messaging.service` (different package). Changed
   to `public` to allow construction — follows `Order` entity precedent where `new Order()` is
   used freely from sibling packages.

2. **Test adaptation — `AdminWebTestBase` real API vs plan skeleton**: Plan test used
   `.with(loginAsOwner())` (assuming `loginAsOwner()` returns `RequestPostProcessor`). Real
   implementation has `loginAsOwner()` returning void and `mockMvc()` as a session-aware wrapper.
   Tests rewritten to match actual base class: `loginAsOwner()` call + `mockMvc().perform(...)`.

3. **Test isolation — `@BeforeEach` cleanup**: `getListReturnsSeededTemplates` expects exactly 4
   (V006 seeds). Since the Spring context is shared across tests in this class, templates created
   by `patchUpdatesSubject` and `deleteSetsActiveFalse` persisted into subsequent test runs.
   Added `@BeforeEach cleanNonSeedTemplates()` that deletes non-seed-named rows and restores
   `active=true` on seed rows, ensuring each test starts from the canonical 4-row state.

4. **`MessagingExceptionAdvice` naming**: Plan uses `MessagingExceptionAdvice`; existing client/order
   advice files use `*ExceptionHandler`. Used plan name since this is a new file in the messaging
   package — no naming conflict.

5. **Structured logging in service**: `TemplateService` logs at INFO with
   `op=template.{create,update,delete} outcome=ok name={}`. Controller logs read-path at INFO with
   `op=template.{list,get} actor={} outcome=ok`.
