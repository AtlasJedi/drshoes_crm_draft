## Decisions / deviations from plan

1. **Attachment field: `att.storageKey()` not `att.name()`**
   Plan body used `att.name()` in the gateway. The actual `Attachment` record (confirmed in 4-3
   dispatch log) uses `storageKey()`. Gateway uses `att.storageKey()` throughout, consistent
   with `PostmarkPayloadMapper`.

2. **`isNetworkError()` helper added — deviation from strict `ResourceAccessException` catch**
   Plan specified: retry only on `ResourceAccessException`. In practice, `SimpleClientHttpRequestFactory`
   (HttpURLConnection) wraps `java.net.SocketException` as bare `RestClientException` during
   header-read phase (before status code is available), not `ResourceAccessException`. Added
   `isNetworkError(RestClientException e)` that returns true for `ResourceAccessException` OR
   any `RestClientException` whose cause chain contains an `IOException`. This preserves the
   retry-on-network / no-retry-on-HTTP intent while handling the transport quirk.
   **Reviewer should verify**: this helper does NOT trigger on `HttpClientErrorException` /
   `HttpServerErrorException` (those extend `RestClientResponseException`, not directly `RestClientException`,
   and their cause chain contains no `IOException`).

3. **`PostmarkEmailGateway` is 157 LOC vs 120 cap**
   The `isNetworkError()` helper + Javadoc on the network-error behaviour added ~30 lines.
   No clean split: all methods share `restClient`, `props`, `blobStorage`. Helper is small
   and directly supports the retry contract. Flagged here for reviewer.

4. **`spring-boot-starter-web` added to email-gateway pom**
   `RestClient` is in `spring-webmvc` which is part of `spring-boot-starter-web`. Without this
   dep, `RestClient` and `SimpleClientHttpRequestFactory` are not on the email-gateway classpath
   (only `spring-boot-autoconfigure` was there before). Alternative: add `spring-webmvc` directly;
   chose `starter-web` since it's the canonical way to pull RestClient support.

5. **`NoOpBlobStorage` placed in storage main sources**
   The IT test imports `com.drshoes.lib.storage.NoOpBlobStorage`. No existing no-op existed.
   Created in storage main sources (not test-jar) so it's available on the normal compile
   classpath without requiring test-jar dependency setup. Acceptable for a utility class that
   is genuinely useful in integration testing across modules.

6. **Test uses `SimpleClientHttpRequestFactory` explicitly**
   Plan IT template used `RestClient.builder().baseUrl(...)` without a factory. The JDK default
   `HttpClient` prefers HTTP/2 which WireMock standalone (Jetty 11) does not support, causing
   all tests to fail with RST_STREAM. Fixed by setting `SimpleClientHttpRequestFactory` (HTTP/1.1)
   in test setUp. The production autoconfig already uses `SimpleClientHttpRequestFactory`.

## Concerns for reviewer (Stage 2)

1. **`isNetworkError()` scope**: Verify it cannot inadvertently catch `HttpClientErrorException`
   (4xx) or `HttpServerErrorException` (5xx). Both extend `RestClientResponseException` which
   extends `RestClientException`. Their cause chain wraps an `HttpMessageConversionException`
   or similar — no `IOException`. The `onStatus` lambda in `doPost` suppresses the throw so
   4xx/5xx flow through `toEntity` and never throw `RestClientResponseException` at all.
   But reviewer should confirm the lambda suppression is correct and the onStatus signature
   matches Spring Boot 3.4's `RestClient` API.

2. **`PostmarkEmailGateway` LOC**: 157 vs 120 cap. Reviewer can suggest a split if warranted
   (e.g. `PostmarkNetworkSender` holding `doPost` + `executeWithRetry`).

3. **Thread-safety**: `RestClient` is documented as thread-safe (immutable request builder
   pattern). `PostmarkProperties` is read-only after construction. `BlobStorage` must be
   thread-safe (S3 client underneath is). No mutable state in gateway. Reviewer to confirm.

4. **Retry semantics**: exactly one retry on network errors, no retry on HTTP errors. Confirmed
   by tests 3+4 (one call each) and tests 5+6 (two calls each). `wm.verify(N, ...)` assertions
   are the primary evidence.

5. **Deferred: attachment display name**: `NoOpBlobStorage.get()` returns empty bytes, so
   attachments in tests have zero-length content. The mapper uses `att.storageKey()` as
   Postmark `Name` field (storage key, not human-friendly filename). This is the known trade-off
   documented in 4-3 dispatch log. No change made here.
