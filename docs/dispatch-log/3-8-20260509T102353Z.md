## Decisions / deviations from plan

1. **Import name**: Plan uses `apiClient` but project convention is `api` (named export from `@/lib/api`). Used `api` to match `lib/orders/api.ts` and `lib/messaging/api.ts`.

2. **Path prefix**: Plan shows paths as `/api/admin/...` but `BASE` env already includes the API base URL. Used `/admin/...` to match established convention in orders and messaging API modules.

3. **postFormData implementation**: Plan showed a simple wrapper delegating to a shared `request()` helper. However `request()` unconditionally sets `Content-Type: application/json` when a body is present — which would override the browser's multipart boundary. `postFormData` calls `fetch` directly, mirroring the same 401 redirect and error-handling logic from `request()`. This is the correct approach to avoid corrupting the multipart content-type header.
