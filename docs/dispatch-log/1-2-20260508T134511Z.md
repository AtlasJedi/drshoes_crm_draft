## Decisions / deviations from plan

1. **Extra test added:** Plan specified 4 unit tests. Added a 5th: `createWithNeitherPhoneNorEmailThrows` to cover the `client_contact_present` validation required by the dispatch instructions (business rule: domain exception before DB CHECK fires). This is the most important behavioral invariant for the service and deserved explicit coverage.

2. **`ClientContactMissingException` added:** Not in the plan's Step 5 code, but mandated by the dispatch instructions. Placed in `com.drshoes.app.client` package alongside `ClientNotFoundException`.

3. **`update()` also validates contact invariant:** The plan's Step 5 code doesn't include contact validation in `update()`, but the invariant applies to any state-changing op. After merge the effective phone/email (existing value overridden by req value) is checked before writing. This is defensive and consistent.

4. **`CreateClientRequest.phone` size max 40:** Plan says `@Size(max = 32)` but the DB column is `VARCHAR(40)`. Used 40 to match schema. Similarly for `UpdateClientRequest`.

5. **`lastName` not `@NotBlank` in `CreateClientRequest`:** The schema has `last_name` as nullable; plan's DTO incorrectly annotated it `@NotBlank`. Removed `@NotBlank` to match the nullable schema column.

6. **`ClientSearchResult.fullName` null-safe:** Added null guard for `lastName` since it is nullable in the schema. The plan's `of()` code didn't handle this.
