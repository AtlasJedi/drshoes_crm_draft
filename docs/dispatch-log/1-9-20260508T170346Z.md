## Decisions / deviations from plan

### Plan-body code is inapplicable (flagged in dispatch prompt)
The plan body calls `log.getEntityType()`, `log.getOperation()`, `parseDelta(log)` etc.
None of these fields exist on `AuditLog`. Dispatcher uses `(method, path, status)` pattern
matching instead — as directed in the dispatch prompt errata block.

### ORDER_UPDATED kind added (not in original 8-kind plan)
A generic PATCH on `/api/admin/orders/{uuid}` (no `/status` or `/items` suffix) cannot be
more specifically classified without body capture. Added `ORDER_UPDATED` as 9th kind
to represent this case. Total: 9 kinds in enum.

### ASSIGNEE_CHANGED / PICKUP_DATE_CHANGED reserved
Both are in the enum but never emitted in M1 — indistinguishable from `ORDER_UPDATED`
without field-level diff. M2 body-capture work will populate them.

### UUID extraction
Used `Pattern.compile("/api/admin/orders/([0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12})(?:/.*)?$")`.
A malformed UUID (e.g. `not-a-uuid`) does not match the pattern → `Optional.empty()` returned.
This is verified by the `malformedUuidInPath_returnsEmpty` test.

### HTTP item-op rows skipped
`POST /api/admin/orders/{id}/items` rows produce `Optional.empty()` — only the INTERNAL row
from `@Audited` is emitted, avoiding double-counting. Verified by `httpItemOpRow_isSkipped_toAvoidDoubleCount`.

### AuditLog missing getters (unplanned fix)
The entity had only setters for `method`, `path`, `status`, `actorId`. Added 4 getters.
Minor fix — inline, not a separate task.

### Curator LOC: 140 vs 120 target
140 total LOC, ~90 net logic LOC. Overage is entirely class-level Javadoc (two-row
semantics, URL prefix assumption, logging rationale). Within reasonable tolerance.

### Logging: none in curator
Curator lives in a hot read path (called per page load for timeline renders).
Emitting even DEBUG would flood. No logging wired here; callers may add DEBUG if needed.
