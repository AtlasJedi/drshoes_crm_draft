## 3. Optimistic-Locking Correctness

**File:** `backend/app/src/test/java/com/drshoes/app/order/OrderServiceIntegrationTest.java`
**Test:** `changeStatus_staleVersion_throwsVersionConflict` (lines 149-167)

The test:
1. Creates an order (version=0).
2. Calls `changeStatus(id, W_REALIZACJI, staleVersion=0)` — this commits, bumping DB version to 1.
3. Calls `changeStatus(id, GOTOWE_DO_ODBIORU, staleVersion=0)` again with the old version — expects `OrderVersionConflictException`.
4. Asserts that the caught exception reports `currentVersion=1`.

**Does this actually prove `@Version` is doing its job?**

Almost — but there is a subtle correctness concern. The conflict check in `OrderService.java` lines 109-110 is:

```java
if (o.getVersion() != req.expectedVersion())
    throw new OrderVersionConflictException(id, o.getVersion());
```

This is a **manual integer comparison in application code**, not a JPA `@Version` optimistic-lock exception (i.e., `OptimisticLockException` / `ObjectOptimisticLockingFailureException`). The `@Version` field on the entity (`Order.java` line 82-84) does provide real DB-level CAS behavior on `UPDATE ... WHERE version = ?`, but the test as written would **pass even if `@Version` were removed from the entity**, because the guard fires before the `save()` call in all cases. The test proves the application-level guard works; it does not prove the `@Version` column prevents concurrent dirty writes at the database level.

**Verdict:** The manual guard is correct and safe for the tested sequential scenario. However, for a production optimistic-lock story the `@Version` annotation is still valuable as a backstop against true concurrent writes. The test comment at line 21 claims `EntityManager.clear()` was avoided because each `@Transactional` call opens its own session — this reasoning is correct for the sequential test case. The architectural decision is sound, but worth noting that the test does not cover the JPA-layer OLE, only the application-layer check.

## 8. Plan Ambiguity — Initial Status (PRZYJETE vs WSTEPNIE_PRZYJETE)

Evidence:
- Plan body line 1070: *"status defaults to PRZYJETE"* (explicit in the test spec bullet point)
- Plan body code sketch line 1112: `o.setStatus(OrderStatus.PRZYJETE);`
- `order_` V001 schema CHECK constraint includes `WSTEPNIE_PRZYJETE` as a valid status value, indicating it is a pre-acceptance state
- Dispatch prompt sanity check (dispatch log line 46) says WSTEPNIE_PRZYJETE

The plan body is internally consistent: both the test spec (line 1070) and the code sketch (line 1112) say PRZYJETE. The dispatch prompt appears to have been out of sync with the plan body on this point. The subagent's choice of PRZYJETE matches the authoritative plan body.

**However, this is a product decision that affects real UX:** PRZYJETE means "received/accepted" while WSTEPNIE_PRZYJETE means "provisionally accepted." For a shoe-repair workflow, the meaningful default depends on whether the initial intake step implies confirmation. The design spec at `docs/superpowers/specs/2026-05-08-milestone-01-orders-design.md` may have the authoritative answer.

**Recommendation:** Accept PRZYJETE per plan body. Flag for human owner confirmation — if the product intent is that orders start in a provisional state, the default must be changed before the controller is built (Task 1-7). Changing it later means a migration or a backfill. **This warrants a one-line owner confirmation before Task 1-7 is dispatched.**
