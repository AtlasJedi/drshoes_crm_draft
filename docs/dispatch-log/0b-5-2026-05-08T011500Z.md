## Decisions / deviations from plan

- **Bucket key:** IP only (plan Task 5 specifies IP; username keying deferred to Task 6 AuthService integration).
- **Capacity:** 5 attempts.
- **Refill rate:** interval refill — full capacity (5) restored after 15-minute window elapses (not greedy/steady; matches plan spec "5 attempts / 15min window").
- **Response on throttle:** `tryConsume` returns `false` — callers (future AuthService, Task 6) map this to HTTP 429. Task 5 is a pure service layer; no HTTP concern here.
- **Clock injection for tests:** Plan does not specify a refill test. Added `bucket_refills_after_window_elapses` (dispatch mandate requires test case (c)). Used `LocalBucketBuilder.withCustomTimePrecision(TimeMeter)` with an `AtomicLong`-backed `TimeMeter` — no Thread.sleep, fully deterministic. The `LoginThrottle` constructor is overloaded: 2-arg (production, `TimeMeter.SYSTEM_NANOTIME`) and 3-arg (testable, injectable).
- **Added `forceAddTokens` alternative considered:** Chose `TimeMeter` injection instead — cleaner, avoids mutating production bucket state in tests, and correctly exercises the actual refill logic rather than bypassing it.
- **Logging:** DEBUG for allowed (high-frequency path), INFO for throttled (actionable security signal). Uses SLF4J structured key=value, no string concatenation.
