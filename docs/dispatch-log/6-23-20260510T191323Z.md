## Decisions

1. **Design-block override (owner directive):** Plan gates on `handoff/design/m6-row-quick-actions-menu.html`. Owner explicitly overrode this gate for the entire dashboard family before dispatch. Visual rules applied from the token table in the dispatch prompt instead.

2. **`StatusChangeConfirm` → `StatusChangeTriggerDialog`:** Plan referenced a fictional `StatusChangeConfirm`. Reality is `StatusChangeTriggerDialog` (M2 component). Props differ: `fromStatus`/`toStatus`/`orderId`/`triggerPreview`/`onConfirm(sendTriggers: boolean)`. Implementation and tests adapted to match reality. No new component created.

3. **Two-step status flow:** `StatusChangeTriggerDialog` requires a non-null `toStatus`. Added an intermediate `"status-pick"` overlay with a `<select>` for the target status before showing the confirm dialog. This is the approach described in the plan's "Note on status selection flow" — inline picker step before the confirm. `StatusChangeTriggerDialog` stays under 120 LOC so no `StatusSelectRow.tsx` extraction needed.

4. **`busy` state removed:** Plan's reference implementation included a `busy` flag, but `StatusChangeTriggerDialog` has no `busy`/`disabled` prop to consume it. ESLint flagged it as unused. Removed — the dialog's own cancel closes on error.

5. **`userEvent` instead of `fireEvent`:** Radix DropdownMenu requires the full pointer event sequence (pointerdown, pointerup, click) to open its content portal. `fireEvent.click` alone is insufficient in jsdom. Tests use `userEvent.setup()` + `await user.click()` which fires the complete sequence. All 5 tests pass.

6. **New dep — `@radix-ui/react-dropdown-menu`:** Not present in `apps/web/package.json`. Installed (14 packages). Consistent with existing Radix usage (`@radix-ui/react-dialog`). Lockfile updated and included in the feature commit per plan instructions.

7. **`WSTEPNIE_PRIJETE` excluded from picker:** Domain convention (reserved for PUBLIC_INTAKE source). Not exposed in manual status-change UI.
