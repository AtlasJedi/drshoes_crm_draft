## SQL Correctness Notes

1. **`orderCount` accuracy** — `countByClientIdAndDeletedAtIsNull` uses explicit JPQL `AND o.deletedAt IS NULL`. Correct. IT test `summaryExcludesSoftDeletedOrders` seeds 2 soft-deleted orders and asserts `orderCount=0`. PASS.

2. **`openOrderCount` correctness** — computed as `total - closed` where `closed = countByClientIdAndStatusInAndDeletedAtIsNull(clientId, [WYDANE, ANULOWANE])`. JPQL predicates include both `AND o.deletedAt IS NULL` and `AND o.status IN :statuses`. CLOSED_STATUSES = `List.of(OrderStatus.WYDANE, OrderStatus.ANULOWANE)` — both statuses present. IT happy-path test seeds 1 WYDANE + 3 active and asserts `openOrderCount=3`. Subtraction math `Math.max(0L, total - closed)` guards against impossible negative. PASS.

3. **`lastOrderAt` / Hibernate LIMIT** — Spring Boot 3.4 ships with Hibernate 6.x. JPQL `LIMIT 1` is valid in HQL/JPQL on Hibernate 6+. Query uses `o.receivedAt ORDER BY o.receivedAt DESC LIMIT 1`. The field is `receivedAt` (entity name), not `created_at` (column). Plan §3.4 says "created_at of most-recent order" but the dispatch log notes `receivedAt` is the meaningful "placed" timestamp in the entity, which is consistent with how the rest of the codebase treats it. The discrepancy is nominal — the implementer made the correct pragmatic choice. IT test seeds `receivedAt` and asserts `lastOrderAt` is non-empty. PASS.

4. **`unreadThreadCount` correctness** — `countByClientIdAndDiscardedAtIsNullAndUnreadCountGreaterThan(clientId, 0)` is the method called (verified in `ClientSummaryService.getSummary`). Spring Data derives `WHERE client_id = ? AND discarded_at IS NULL AND unread_count > ?`. Both predicates present. IT happy-path seeds 1 discarded unread thread (excluded) + 1 non-discarded unread (counted) + 1 read non-discarded (excluded by unreadCount=0). Asserts `unreadThreadCount=1`. PASS.

5. **RBAC** — `@PreAuthorize("hasAnyRole('OWNER','EMPLOYEE')")` at class-level on `ClientSummaryController`. IT test `summaryReturns401ForUnauthenticated` asserts 401 for anonymous. PASS.

6. **404 path** — `clientRepo.findById().filter(c.getDeletedAt() == null).orElseThrow(ClientNotFoundException)`. IT test `summaryReturns404ForUnknownClient` uses a zero-UUID and expects 404. Unit test `summaryThrowsForSoftDeletedClient` covers the soft-deleted client path. PASS.

7. **Structured logging** — Service logs `op=getClientSummary clientId={} orderCount={} openOrderCount={} unreadThreadCount={} outcome=ok`. Controller logs `op=getClientSummary actor={} clientId={}` on entry and `...outcome=ok` on exit. Fields `op`, `actor`, `clientId`, `outcome` all present per protocol §7. PASS.

8. **LOC ceiling** — `ClientSummaryService` = 64 LOC (well under 120). `ClientSummaryController` = 49 LOC. PASS.

## Decisions / Observations

- `receivedAt` vs `created_at` discrepancy: acceptable. The plan wording is aspirational; the entity's `receivedAt` is the correct business timestamp.
- Controller logs twice (entry + exit) — minor verbosity, not a defect. Consistent with other controllers in the project.
- `findLastOrderCreatedAtByClientId` JPQL also implicitly excludes non-null `receivedAt` values via `ORDER BY receivedAt DESC` — if `receivedAt` is NULL for an order, it sorts last, but since all orders in tests set `receivedAt`, this is not a practical issue.
