## Decisions / deviations from plan

**1. OrderService.changeStatus signature — uses ChangeStatusRequest record, not (orderId, status, version, actorId).**
The plan's test skeleton assumed `statusService.changeStatus(orderId, "PRZYJETE", 0L, ownerId())` (4 args).
Actual: `orderService.changeStatus(UUID id, ChangeStatusRequest req)` where `ChangeStatusRequest(OrderStatus targetStatus, int expectedVersion)`.
Adapted test to use `orderService.changeStatus(orderId, new ChangeStatusRequest(OrderStatus.PRZYJETE, 0))`.
`version` is `int` on both `Order.getVersion()` and `ChangeStatusRequest.expectedVersion()` — not `long`.

**2. fromStatus / toStatus extraction in OrderService.changeStatus.**
`fromStatus` is captured as `OrderStatus old = o.getStatus()` (already in the method before mutation).
`toStatus` is `req.targetStatus()`. Both converted to string via `.name()` before calling `triggerEngine.onStatusChange`.
Call site: `triggerEngine.onStatusChange(saved.getId(), old.name(), req.targetStatus().name())` — placed after `orderRepo.save(o)` and before the return, inside the `@Transactional` boundary.

**3. afterCommit + REQUIRED propagation — stale transaction context bug.**
The plan stated that `sendForTrigger`'s `@Transactional(REQUIRED)` would open a new transaction in the `afterCommit` callback. This is incorrect in practice: Spring's `afterCommit` runs while the `ThreadLocal` transaction context is still bound (status=committed). `REQUIRED` tries to join the existing (completed) context and fails with `InvalidDataAccessApiUsageException: no transaction is in progress`.

**Fix:** `TriggerEngine` injects `PlatformTransactionManager` and builds a `TransactionTemplate` with `PROPAGATION_REQUIRES_NEW`. Each `fireOne` call from the `afterCommit` path is wrapped in `requiresNewTx.executeWithoutResult(...)`, which suspends the stale context and starts a genuine new transaction. `sendForTrigger`'s own `@Transactional` then participates in that new transaction normally.

`fireScheduled` (called from a scheduled context with no stale tx) calls `fireOne` directly — no wrapping needed.

**4. ClientRepository not injected in TriggerEngine.**
The plan skeleton included `ClientRepository` as a TriggerEngine dependency. Not needed: `fireOne` gets `clientId` from `order.getClientId()` (raw UUID field). The router's `sendForTrigger` accepts `clientId` directly. Removed to keep the constructor lean.

**5. Test helpers — no OrderStatusService, no ownerId() method.**
Plan test used `OrderStatusService` (doesn't exist) and `ownerId()`. Replaced with direct `OrderService.changeStatus` calls. Client+order rows inserted via JdbcTemplate (same pattern as MessageRouterIntegrationTest). No `ownerId` parameter needed — `changeStatus` doesn't require an actor id.

## Concerns
- **LOC budget:** TriggerEngine is ~130 LOC including Javadoc. Slightly over the 120 plan target but not meaningfully splittable — all logic is cohesive. Documented exception analogous to TimelineEventCurator (1-9).
- **PROPAGATION_REQUIRES_NEW in afterCommit:** This is the correct pattern for post-commit side effects. Each trigger fires in its own transaction; a failure in one channel does not prevent others from being attempted (caught by the per-trigger `try/catch`). The `IdempotencyService.claimTriggerFire` itself also uses `REQUIRES_NEW` — nested REQUIRES_NEW transactions are supported by Spring's `JpaTransactionManager`.
