## Decisions / deviations from plan

- **V001 proto table dropped**: V001 included a prototype `photo` table with a different
  schema (VARCHAR(255) s3_key, nullable uploaded_by, CHECK label, extra width/height/bytes/position
  columns, no size_bytes/original_filename). V009 drops it with `DROP TABLE IF EXISTS photo` before
  recreating with the authoritative M3 schema. No production data existed.

- **Photo @Entity stub + PhotoLabel enum + PhotoRepository shipped in 3-1** (not deferred to 3-5):
  The @Disabled PhotoRepositoryTest references Photo/PhotoLabel/PhotoRepository. Java compiles the
  test body regardless of @Disabled, so the symbols must exist. Stubs are added to the main source
  tree so Spring Boot context loads cleanly. Task 3-5 replaces Photo.java with the full entity
  (no other changes needed to PhotoLabel or PhotoRepository, which are already final).

- **Photo.id uses @GeneratedValue**: V009 declares `id UUID PRIMARY KEY` without a DB-level
  DEFAULT. Hibernate's @GeneratedValue generates the UUID application-side before INSERT,
  consistent with how Order.id is handled (also @GeneratedValue, DB default is used only at
  DB-level if bypassed).

- **newClient() helper**: Plan body called `c.setFullName(...)` but Client entity has
  `setFirstName`/`setLastName` (separate columns per V001). Fixed to `setFirstName("Klient") +
  setLastName("Testowy")`.

- **newOrder() helper**: Plan body called `o.setOrderType(OrderType.NAPRAWA)` but Order entity
  has no orderType field (not in V001 schema). Removed. Order is created with clientId + status
  only, which satisfies the NOT NULL constraints.
