## Audit findings (8 required points)

### Point 1 — UUID-skip regex `[0-9a-f\\-]{36}` in `extractEntityType`

**Finding: FOLLOW-UP (not blocking).**

The regex is **not anchored** (no `^...$`) and is **lowercase-only** (`[0-9a-f]` misses `A-F`). Tested with real paths:

- `/api/admin/orders/{lowercase-uuid}` → `"orders"` ✅ (correct skip)
- `/api/admin/orders/{UPPERCASE-UUID}` → `"550E8400-E29B-41D4-A716-446655440000"` ✅ (wrong — UUID not skipped, UUID segment returned as entity type)
- `/api/admin/orders/{uuid}/items` → `"items"` ✅
- `/api/admin/orders/{uuid}/status` → `"status"` ✅
- `/api/admin/photos/{uuid}/file` → `"file"` ✅
- `/api/admin/messages/{uuid}/retry` → `"retry"` ✅
- `/api/admin/auth/login` → `"login"` ✅

**Severity: LOW / non-blocking.** Uppercase UUIDs cannot appear in real request URIs in this stack: Java's `UUID.toString()` always produces lowercase, and frontend `crypto.randomUUID()` is also lowercase per spec. Spring path-variable binding validates UUID format but does not case-normalize the URI string. In practice, `getRequestURI()` will always carry lowercase UUIDs. The regex works correctly for all real traffic.

The lack of anchoring (`{36}` with no `^..$`) means a longer string like `550e8400-e29b-41d4-a716-4466554400001` (37 chars) would NOT be skipped — but real UUIDs are exactly 36 chars so this is also non-triggering.

Follow-up: fix to `(?i)[0-9a-f-]{36}` or use `UUID.fromString(p)` try/catch for robustness. Low priority — deferred.

### Point 2 — `writeWithSpan` exception propagation

**Finding: CORRECT.**

Flow in `AuditSpanHelper.writeWithSpan`:
```java
try (Scope ignored = span.makeCurrent()) {
    action.run();  // may throw RuntimeException/Error
} finally {
    span.end();    // always runs; span ends before exception propagates
}
```

If `action.run()` throws, the `try-with-resources` closes `Scope`, then `finally` ends the span, and the exception propagates out of `writeWithSpan`. In `AuditWriteCoordinator.persistHttp/persistAnnotated`, the call to `spanHelper.writeWithSpan(...)` is inside `try { ... } catch (Exception ex) { log.warn(...) }` — so the exception is caught and logged, never double-caught or masked. Exception path is: `action.run()` throws → `span.end()` in finally → propagates out of `writeWithSpan` → caught in coordinator's `catch(Exception)` → WARN logged, caller unaffected. Clean single catch path. No double-catch issue. ✅

### Point 3 — `ATTR_ENTITY_ID = "null"` string when entityId is null

**Finding: ACCEPTABLE (style call, OTel-compliant).**

OTel Java SDK prohibits null attribute values — `AttributeKey.stringKey(...).set(null)` throws `NullPointerException`. Stage 1 uses `entityId != null ? entityId : "null"` which satisfies the OTel constraint by passing a non-null string. Alternative: omit the attribute when null (`SpanBuilder.setAttribute` can be conditionally called). Both are valid; `"null"` string is a common OTel convention for absent optional attributes. No correctness issue. ✅

### Point 7 — trace_id with no active span → invalid trace ID → null

**Finding: CORRECT.** ✅

`AuditSpanHelper.currentTraceId()`:
```java
SpanContext ctx = Span.current().getSpanContext();
if (!ctx.isValid()) return null;
```

`Span.current()` when no span is active returns the OTel NOOP span. Its `SpanContext.isValid()` returns `false` because both traceId and spanId are all-zeros (`"00000000000000000000000000000000"`). Thus `!ctx.isValid()` is `true` → `null` returned → `writer.write(..., null)` → `trace_id IS NULL` in the row.

The integration test `trace_id_is_null_when_no_active_span` confirms this path asserts `trace_id IS NULL`. The inverse test confirms non-null under an active span. Both pass. ✅
