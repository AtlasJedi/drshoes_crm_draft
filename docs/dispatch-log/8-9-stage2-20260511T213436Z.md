## Audit Findings (10 points)

### 1. Idempotency scope — single `count() >= 6` guard
**ACCEPTABLE.** `DemoSeedRunner.run()` checks `clientRepository.count() >= 6` and short-circuits the
entire run (clients + orders + threads) in one guard. This is intentional single-gate semantics for
the demo seed use case. The plan does not mandate per-factory idempotency; it specifies "duplicate-seed
safety" broadly. The `seedIsIdempotent` integration test verifies the guard works correctly.
**Accepted risk:** if a real operator creates 6+ clients but no seed orders exist, the runner will skip
seeding orders too. This is documented in Stage 1's follow-ups and is acceptable for the demo-local scope
(`@Profile("local")` guard ensures prod is never touched). No fixup required.

### 2. WSTEPNIE_PRZYJETE backward transition (PRZYJETE → WSTEPNIE_PRZYJETE)
**CONFIRMED INTENTIONAL — FREE TRANSITIONS.** `OrderService.changeStatus` (line 117–131) has exactly
one validation: `o.getVersion() != req.expectedVersion()` (optimistic locking). There is NO state-machine
transition guard. Any `targetStatus` is accepted. This matches the project-wide design principle: free
status transitions with trigger preview. PRZYJETE → WSTEPNIE_PRZYJETE succeeds correctly. No
downstream invariants are violated: audit still writes, trigger suppressed via `sendTriggers=FALSE`.

### 3. `picked_up_at` mechanism on WYDANE orders
**CONFIRMED CORRECT.** `OrderService.changeStatus` lines 123–124:
```java
if (req.targetStatus() == OrderStatus.WYDANE && o.getPickedUpAt() == null)
    o.setPickedUpAt(Instant.now());
```
The seed does NOT set `pickedUpAt` manually — it is set automatically by `changeStatus`.
**Known limitation:** `Instant.now()` is used, not the seed's historical `receivedAt + X days`. For
a demo, this means WYDANE orders show a `picked_up_at` of the seed run moment, not a realistic past
date. This is acceptable for demo purposes and matches the plan's intent. Tracked as post-M8 polish if
needed.

### 4. Audit under non-HTTP context
**SAFE — NO NPE RISK.** Two independent reasons:
(a) `AuditLogAspect` HTTP pointcut is `execution(public * com.drshoes.app..api..*Controller.(..))`.
    Seed runs via `ApplicationRunner` at startup, not through a controller. The HTTP aspect NEVER fires.
(b) `AuditWriteCoordinator.persistHttp` has `if (attrs == null) return;` at line 41 — safe even if
    called with null `ServletRequestAttributes`.
(c) `OrderService.create` internally calls `itemService.addItem(savedId, ir)` directly on `OrderItemService`
    — NOT the `@Audited`-wrapped `OrderService.addItem` method. Spring AOP self-invocation rule: the `@Audited`
    pointcut does NOT fire for the items created during order seeding. No audit rows are generated for
    seed data. This means demo dashboard "recent activity" will not show seed-era entries, which is
    acceptable (and arguably desirable — seed data should not pollute audit timeline). No NPE risk.

### 5. Photo seed absence

are intentionally omitted from the service-layer seed path." Stage 1 correctly deferred this and
documented it in commit message + dispatch log. The deferral is explicitly sanctioned by the plan.
One follow-up: the plan note says "tracked as post-M8 hygiene" — Stage 1 captured this correctly.

## Plan Errata (5 items — document for future tasks)

1. **`UpdateClientRequest` arity is 7, not 8.** Plan §2034 shows 8-arg call
   `(null, null, null, null, channel, null, rodoConsent, null)`. Actual record:
   `(firstName, lastName, phone, email, preferredChannel, rodoConsent, notes)` — 7 fields.
   Correct call: `(null, null, null, null, channel, rodoConsent ? Boolean.TRUE : null, null)`.

2. **`CreateOrderItemRequest` arity is 4, not 5.** Plan §2033 shows
   `new CreateOrderItemRequest(kind, description, null, 0, 0)` — 5 args. Actual record:
   `(kind, description, craftsmanNotes, priceCents)` — 4 fields. No `quantity` field exists.

3. **`ChangeStatusRequest.expectedVersion` is `int`, not `long`.** Plan §2035 uses `long v = version`.
   Actual record field is `int expectedVersion`. `OrderDto.version()` also returns `int`.

4. **`ClientDto.id()` is `UUID`, not `String`.** Plan's factory used `UUID.fromString(dto.id())`.
   Since `ClientDto.id()` is already `UUID`, no conversion needed.

5. **`WSTEPNIE_PRZYJETE` not `WSTEPNIE_PRIJETE`.** Plan §2037 references the status without verifying
   spelling. Actual enum value in `OrderStatus.java` is `WSTEPNIE_PRZYJETE` (with Z after R).

## Decisions

1. **Idempotency granularity accepted.** Single `clientRepository.count() >= 6` guard is correct for
   the demo-seed use case. Per-factory guards would add complexity without meaningful safety benefit
   in a `@Profile("local")`-only component. Documented as post-M8 optional enhancement.

2. **Photo seed deferral confirmed.** Plan explicitly sanctions JPA-bypass or omission when MinIO is
   unavailable. No fixup needed. Follow-up item for post-M8.

3. **`picked_up_at = Instant.now()` for seeded WYDANE orders.** Acceptable for demo. Not a bug —
   the mechanism is correct; only the timestamp realism is sub-optimal. No fixup needed.

4. **No audit rows for seed data.** Confirmed safe and intentional: HTTP pointcut won't fire outside
   a controller invocation, and `@Audited` wrapper methods are not called via seed path. Dashboard
   "recent activity" showing empty for seed data is correct behavior.
