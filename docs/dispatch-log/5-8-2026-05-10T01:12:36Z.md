## Decisions

### Service split at LOC cap
`MessageThreadService` was ~35 LOC. Adding ~120 LOC of mutation methods would exceed the 120-LOC cap. Split per plan: read-side stays in `MessageThreadService`, write-side goes into new `MessageThreadMutationService`. Both stay in `com.drshoes.app.messaging.service`.

### Back-compat overload retained
`MessageThreadService.findOrCreateForClient(UUID)` retained as a one-arg back-compat overload delegating to `findOrCreateForClient(UUID, "EMAIL")`. Verified callers: `MessageRouter` calls the one-arg form at lines 104 and 146 â€” covered. `InboundMessageService` calls the two-arg form directly â€” covered. No call-site changes required.

### Repository finder method choice
The plan specifies `findFirstByClientIdAndChannelOrderByCreatedAtAsc` and `findFirstByRawSenderAndChannelOrderByCreatedAtAsc` â€” simpler variants without the discard filter. These coexist with the existing filtered variants used by InboundMessageService. The mutation-service tests mock these simpler methods (matching the plan spec). The existing `findFirstByClientIdAndChannelAndDiscardedAtIsNullOrderByLastMessageAtDesc` is still used by InboundMessageService (unchanged).

### Cross-task integration â€” curator patterns updated atomically
Per the task instructions, the curator `Pattern` constants, `TimelineEventKind` comments, and `TimelineEventCuratorM5Test` path strings were all updated in the same commit as the service split. This ensures the audit row path `MessageThreadMutationService#markRead` (etc.) matches the curator patterns from the moment the split ships â€” no window of silent timeline kind loss.

### AuditLogAspect uses `SimpleClassName#methodName` format
Verified at `AuditLogAspect.java:112`: `method.getDeclaringClass().getSimpleName() + "#" + method.getName()`. New paths will be `MessageThreadMutationService#markRead`, `MessageThreadMutationService#assignUnmatched`, `MessageThreadMutationService#discardUnmatched` â€” matching the updated curator patterns.
