# Review: Task 1-8 — V005 + @Audited + AuditLogAspect parent-id population

## 1. Two-Row Architecture

**Two advices confirmed.** `AuditLogAspect` (lines 56-72 and 80-87) contains:
- `audit()` — `@Around` on `execution(public * com.drshoes.app..api..*Controller.*(..))` (controller HTTP rows)
- `auditAnnotated()` — `@Around("@annotation(audited)")` (service `@Audited` rows)

These are genuinely separate proxy boundaries. The controller advice fires at the HTTP adapter layer; the `@Audited` advice fires at the Spring service bean proxy. There is no clean way to enrich a controller row from a nested service call without thread-local coupling — the two-pointcut design is architecturally correct and the dispatch log explains the tradeoff well.

**`path NOT NULL` — BLOCKER CONFIRMED.**

`V001__init.sql` line 486: `path VARCHAR(255) NOT NULL`.

`persistAnnotated()` (AuditLogAspect.java line 110) synthesizes a value:
```java
String syntheticPath = method.getDeclaringClass().getSimpleName() + "#" + method.getName();
writer.write("INTERNAL", syntheticPath, 0, null, null, parentId);
```

The `path` column is satisfied — `syntheticPath` will be e.g. `OrderService#addItem`, never null. The `method` column is satisfied by the literal `"INTERNAL"`. The `status` column is satisfied by `0`.

**This is NOT a blocker.** The impl correctly anticipated the NOT NULL constraint and synthesizes meaningful values. The concern is resolved.

However, there is a **MED** issue: `status = 0` is not a valid HTTP status code and will land in `audit_log` for every service row. The timeline curator (1-10) must filter or handle `status=0` service rows differently from HTTP rows. This is undocumented. Recommend adding a constant `INTERNAL_STATUS = 0` or a DB CHECK constraint exemption note. Not a blocker for 1-8 but worth a fixup comment.

## 2. Plan Adherence vs Deviation

The plan (line 1310-1312) says: "when wrapping a method annotated `@Audited(parent = "#orderId")`, evaluate the SpEL against JoinPoint args + return value... set the resulting UUID on the AuditLog row before persisting."

The plan wording implies a single-pointcut enrichment approach (modifying the HTTP controller row). The impl uses a second independent pointcut, which writes a *new* row rather than enriching the HTTP row. This is a **sound deviation**: enriching the HTTP row from within a nested service call requires thread-local or `RequestAttributes`-based state passing between two different proxy levels, which is fragile and breaks test isolation. The two-row design is cleaner, fully aligns with the errata's intent (parent_entity_id available on audit rows), and the dispatch log documents the rationale explicitly.

The plan errata (lines 37-43) confirms single `parent_entity_id UUID` column + partial index — V005 (lines 4-6) matches exactly.

No unauthorized deviation. The impl subagent made the right architectural call.

## 3. SpEL Sandboxing

**Correctly sandboxed.** `AuditedParentResolver.java` line 62-64:
```java
SimpleEvaluationContext ctx = SimpleEvaluationContext
    .forReadOnlyDataBinding()
    .build();
```

This blocks `T(java.lang.Runtime).getRuntime().exec(...)` and Spring bean references via `@beanName`. Confirmed: NOT `StandardEvaluationContext`.

**Residual concern (LOW):** `SimpleEvaluationContext.forReadOnlyDataBinding()` still permits property navigation on bound variables. If a future annotation uses `@Audited(parent="#req.someField")` where `req` is a complex object with getter-chain side effects, or if a deserialized object has a getter that triggers lazy loading or logs something sensitive, this would navigate through it. The risk is low since `@Audited` is a developer-controlled annotation, not user-controlled input. The SpEL expression is compiled into the class file, not taken from user input at runtime — attacker control over `#orderId` itself is limited to UUID injection, which is safe. The sandbox posture is appropriate.

**No sandbox escape concern for current usage.** Future reviewers should flag any `@Audited(parent = "#req.*")` usage with complex object types.

## 4. Exception Safety

`AuditedParentResolver.resolve()` (lines 46-59):

```java
} catch (Exception e) {
    log.warn(...)
    return null;
}
```

`catch (Exception e)` covers:
- `SpelEvaluationException` (extends `EvaluationException` extends `RuntimeException` extends `Exception`) — covered
- `SpelParseException` (extends `ParseException` extends `RuntimeException` extends `Exception`) — covered
- `ClassCastException` — covered (line 52 uses pattern matching `instanceof UUID u`, not a hard cast)
- `IllegalArgumentException` from `UUID.fromString(result.toString())` on line 53 — covered

**Type coercion is safe:** lines 51-53:
```java
if (result == null) return null;
if (result instanceof UUID u) return u;
return UUID.fromString(result.toString());
```

Pattern matching prevents `ClassCastException`. `UUID.fromString()` failure is caught by the outer `catch (Exception e)`. No unhandled throwable path for normal execution.

**One subtle gap (MED):** `catch (Exception e)` does NOT catch `Error` subclasses (`OutOfMemoryError`, `StackOverflowError`). These are extreme scenarios — a recursive SpEL expression on a deeply-nested object graph could theoretically cause `StackOverflowError`. In practice this is academic for the current usage (`#orderId` is a UUID). No fixup required for 1-8 but worth noting.

Exception safety claim: **verified**.

## 6. TDD Discipline

### Test 1: `auditedAnnotationPopulatesParentEntityId` (line 105-124)
**Strong.** Asserts `latest.getParentEntityId().isEqualTo(orderId)` — exact UUID match, not just "not null". The order UUID is a live value from `orderService.create(...)`, binding the assertion to the actual domain object. This would catch a broken SpEL binding that returned the wrong UUID.

### Test 2: `methodWithoutAuditedAnnotationLeavesParentEntityIdNull` (line 131-146)
**Weak — has a logic gap (MED).** The test calls `clientService.create(...)` directly (no HTTP context). Since there is no HTTP request in scope, `RequestContextHolder.getRequestAttributes()` returns null, `persistHttp()` returns early (line 92: `if (attrs == null) return`), and zero audit rows are written. The loop `rows.forEach(row -> assertThat(row.getParentEntityId()).isNull())` iterates over an empty list and asserts nothing. The test passes vacuously — it proves nothing about the `@Audited` pointcut not firing.

**What the test should do:** Call the test via HTTP (like the existing `login_writes_audit_log_row` test) against a controller endpoint that does NOT trigger `@Audited`, then assert that the resulting HTTP audit row has `parent_entity_id = null`. Alternatively, call `orderService.get(orderId)` (no `@Audited`) with a mock HTTP context. As written the test is a false green.

### Test 3: `spelFailureDoesNotBreakAuditRow` (line 165-190)
**Partial.** The test verifies two things:
1. `AuditedParentResolver.resolve("#nonExistentVariable")` returns null (direct unit call, lines 167-177) — valid isolation test.
2. `orderService.addItem(...)` still writes an audit row (lines 181-189) — this proves the happy path writes a row, but does NOT exercise the broken-SpEL code path through the full aspect pipeline.

The claim "the audit row still writes when SpEL fails" is **not actually proven by this test.** To prove it, the test would need to use a method with `@Audited(parent="#nonExistentVar")` and verify that an audit row was written despite the SpEL failure. There is no `@Audited` method with a broken expression being called through the aspect.

This is the most significant TDD gap. The safety claim is real (the `try/catch` is correct) but the test does not prove it end-to-end.

### Test 4: `spelEvaluatesAgainstArgs` (line 198-209)
**Good.** Direct resolver test, exact UUID match. Provides a CI guard for the `-parameters` flag.
