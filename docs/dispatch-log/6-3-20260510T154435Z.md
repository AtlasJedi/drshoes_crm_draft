## Decisions

1. **Pre-existing test fixup included in same commit:** `MessageThreadRepositoryInboundIntegrationTest.findFirstByClientIdAndChannel_returnsMostRecent` was inserting two active `(clientId, EMAIL)` threads to test "returns most recent" ordering. With V013 this is a `DuplicateKeyException`. The test was updated in-scope (same commit) because it is a direct consequence of V013 and is trivial: one discarded + one active thread still exercises the `discarded_at IS NULL` filter while satisfying the uniqueness constraint. No separate fixup commit warranted.

2. **Deduplication DELETEs:** Included in V013 as the plan specifies, retaining the oldest row per `(channel, client_id)` and `(channel, raw_sender)` respectively. Dev/test DB is always clean; this is for production safety.

3. **`@NotNull` form:** Used fully-qualified `@jakarta.validation.constraints.NotNull` inline rather than adding an import, consistent with the plan's code snippet. The field already had `@Column(nullable = false)` so this aligns JPA validation with the DB constraint.

4. **No `discarded_at IS NULL` predicate on `message_thread_unmatched_idx` from V012:** V012 created a non-unique index `message_thread_unmatched_idx ON (channel, raw_sender) WHERE client_id IS NULL`. V013 adds a separate *unique* partial index `message_thread_unique_unmatched ON (channel, raw_sender) WHERE raw_sender IS NOT NULL AND discarded_at IS NULL`. These coexist correctly; the V012 index is a query-optimization index, V013 is a constraint index.
