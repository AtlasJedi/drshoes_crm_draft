## Deviations from prompt

1. **`MusicContext` exported** (`components/admin/music/MusicProvider.tsx:101`): Prompt specified `MusicSidebarDot` should call `useMusicContext()`. Changed to `useContext(MusicContext)` directly so the component returns null gracefully when rendered outside MusicProvider (i.e., in AdminSidebarNav tests). This is strictly safer and behaviorally equivalent in production where the provider is always present.

2. **`QueueCol` thumbnail uses `<div>` not `<button>`** (`muzyka/components/QueueCol.tsx:118`): Prompt implied thumbnail + meta both clickable as separate buttons. Two buttons with identical `aria-label` caused test failures ("multiple elements"). Fixed: thumbnail uses a `<div aria-hidden>`, only the meta button carries the `aria-label`. Functionally identical from a UX standpoint.

3. **`SearchColV2` play action via thumbnail click** (`muzyka/components/SearchColV2.tsx:100`): The "Odtwórz" action wraps the thumbnail area as a button (matching designer's intent); the existing MusicClient tests expected `findByRole("button", { name: /Title a/ })` which no longer matches the v2 layout (now `aria-label="Odtwórz: Title a"`). Updated tests to use new aria-labels.

4. **`AdminSidebarNav` Muzyka dot wrapping** (`components/admin/AdminSidebarNav.tsx:73`): Designer shows `now-dot` inside the `.side-link` span. Our sidebar uses Next.js `<Link>` for the NavLink — adding the dot inside the link element would require restructuring NavLink. Chose to wrap the NavLink + dot in a `<div className="flex items-center">` instead. Dot still visually adjacent to the link. Fully functional.
