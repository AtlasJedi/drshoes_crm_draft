## Decisions / Surprises

- **Column header renamed from "Zgoda RODO" to "Status RODO":** The plan's test uses `screen.getByText(/zgoda/i)` to find the green pill ("zgoda · MM.yyyy"). With "Zgoda RODO" as the column header, two elements match `/zgoda/i` and `getByText` throws "multiple elements found". Renaming the header to "Status RODO" resolves the collision without changing the pill copy. The visual intent is the same — the column displays RODO consent status.
- **`next/link` mock added to test file:** The plan's test spec did not include the `vi.mock("next/link")` call, but the component uses `Link` for row anchors and pagination. Without the mock, jsdom renders links with incorrect href resolution. The mock renders a plain `<a href={href}>` — consistent with all existing tests that use `next/link` in jsdom (e.g. `page.test.tsx` in calendar). This is a minor implementation detail not a behavior deviation.
- **No `lib/log.ts` import:** This is a pure presentational SC that receives data from its caller. No side effects, no fetches. Consistent with other display-only SCs.
- **RodoInline is a file-private placeholder.** Task 7-10 will create `RodoBadge` as a shared SC — at that point, replace `RodoInline` with `import { RodoBadge } from "@/components/rodo/RodoBadge"` and delete the private helper.
