## Decisions / deviations from plan

1. **Typed `HttpError` instead of string parsing**: Rather than parsing `"API <status>: ..."` strings
   in LoginForm, extended `lib/api.ts` with a typed `HttpError` class that carries `status: number`.
   `resolveLoginError()` in `lib/auth/login-error.ts` instanceof-checks it. This is cleaner, avoids
   fragile regex, and benefits all future callers (0b-12, etc.). The old `throw new Error(...)` path
   is now `throw new HttpError(...)` — same message shape, same log fields, non-breaking.

2. **Extracted `login-error.ts`**: `LoginForm.tsx` would have exceeded 80 LOC with the error resolver
   inlined. Moving it to `lib/auth/login-error.ts` keeps both modules under 80 LOC and is independently
   testable.

3. **`Suspense` wrapper in `page.tsx`**: Next.js 16 requires `useSearchParams()` client components to
   be wrapped in `<Suspense>` at the page boundary, or the build fails with a prerender error. Added
   `<Suspense>` in `login/page.tsx` (no fallback needed — form renders immediately on client).

4. **`router.push` cast**: `typedRoutes: true` is set in `next.config.mjs`. The `?next=` param is a
   runtime string so it can't satisfy `RouteImpl<string>`. Cast via `as AnyRoute` (local `type AnyRoute = any`)
   with an explaining comment — same pattern used elsewhere in the codebase for dynamic redirects.

5. **`aria-invalid` boolean**: When no error is shown, passes `undefined` (omitted) rather than `false`
   to avoid a superfluous attribute in the DOM.

## Logging / metadata wired
- `components/auth/LoginForm.tsx` uses `createLogger("login-form")`
- Logs `op=login outcome=ok` at INFO on success
- Logs `op=login outcome=invalid` at WARN on 400/401
- Logs `op=login outcome=error` at WARN on 429/5xx/network
- Does NOT log email or password — only outcome and (where available) requestId from `HttpError`
