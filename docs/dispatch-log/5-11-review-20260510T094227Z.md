## Failure-mode analysis
- **Gateway down (dispatcher throws):** `@Transactional` on `sendNewToClient` means `saveAndFlush` rolls back if `dispatcher.dispatch` throws unchecked. No orphan message row. Spring MVC returns 500. Acceptable.
- **Client has both email and phone null, channel=EMAIL:** `validateChannelAvailability` fires 422 ("Client has no email address"). Correct тАФ channel validation is in the controller and uses `isBlank()` guard. No gateway call is made.
- **Null channel in request:** as noted above тАФ null channel bypasses `validateChannelAvailability` (neither branch matches null), then `Channel.valueOf(null)` in `sendNewToClient` throws NPE тЖТ 500. This is a gap (should be 400), but it is a carry-forward pattern from 5-10's sendReply and is within spec since the plan does not require a null-channel 400.
- **Race condition тАФ concurrent composes for same client+channel:** `findOrCreateForClient` uses a `findFirstтАжorElseGet(create)` pattern. There is NO UNIQUE constraint on `(client_id, channel)` in `message_thread` (V001 creates an index, not a unique constraint; V012 adds only a CHECK constraint for client_id/raw_sender mutual exclusion). Two concurrent requests for the same clientId+channel can both pass the `findFirst` (returning empty) and both create a new thread row тАФ resulting in duplicate threads for the same client+channel pair. This is a pre-existing risk in `sendReply` and `sendRetry` as well (they call the single-arg overload of `findOrCreateForClient` which also has no unique-constraint safety net), but the risk is newly activated for the compose path where two concurrent operator composes to the same client are plausible. Medium risk тАФ no data corruption (messages on the second thread are not lost), but the expected single-thread-per-client-per-channel invariant breaks. A `UNIQUE INDEX ON message_thread (client_id, channel) WHERE client_id IS NOT NULL` would close this permanently.
- **`findOrCreateForClient` called after channel validation тАФ correct ordering:** validation fires before thread creation. A 422 response does not create a spurious thread row. PASS.
