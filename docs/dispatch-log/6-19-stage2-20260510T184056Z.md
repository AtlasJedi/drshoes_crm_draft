## Review Findings

### 1. Drag flow correctness

Flow is correct. `onDragEnd` uses a functional state updater `setColumns((prev) => { ... })` to atomically capture the pre-move snapshot (`setSnapshot(prev)` inside the updater), apply `applyOptimisticMove`, and derive `pendingMove` from the same `prev` snapshot. All three state writes (`columns`, `snapshot`, `pendingMove`) land in a single React 18 batch since they occur in the same synchronous updater call chain. The `applyOptimisticMove` function is pure (no mutation). Cancel path: `onCancel` reverts `setColumns(snapshot)`, clears `snapshot` and `pendingMove`. Confirm success path: column state stays at optimistic value (no re-read from server in M6 scope, acceptable). Confirm error path: `savedSnapshot` is captured synchronously at the top of `onConfirm` before the `await`, then used in the `catch` block to revert — correct under React 18 semantics. Test coverage confirms all three paths. No issues found.

### 2. Drag-ghost styling vs locked rules

**NEEDS-FIXUP applied.** Ghost overlay: `opacity-90 rotate-1 shadow-lg` on the `DragOverlay` wrapper — matches locked spec. Source card while dragging: `opacity: isDragging ? 0.4 : 1` via `useSortable` in `KanbanCard.tsx` — correct. Cursor: `cursor-grab active:cursor-grabbing` in `KanbanCard.tsx` — correct. Same-status drop: suppressed at both `KanbanBoard.handleDragEnd` (early return) and `useKanbanDnd.onDragEnd` (early return) — correct. **Drop-target column highlight:** `KanbanColumn.tsx` used `bg-black/10` on `isOver`, which violates the locked spec (`ring-2 ring-[var(--pink)] ring-offset-2 ring-offset-transparent`). Fixed in Stage 2 fixup commit. The ring approach is visually superior (doesn't blend into the dark column header), so the fix is also a product improvement.

### 3. Drawer regression

`OrderDrawerStatusChanger.tsx` is correctly migrated. Prop contract from upstream callers is unchanged (`order: OrderDto`, `onOrderUpdated`). Toggle behavior: the drawer now calls `onConfirm={() => { void handleConfirm(); }}`, which ignores the `sendTriggers` boolean — plan-sanctioned for M6 (M7 wires it fully). The removed `busy` state does not regress functionality: the dialog closes synchronously on confirm, so the button-disabled pattern is unnecessary. Conflict handling (409 path → `setConflict(true)`) is preserved. The `previewForStatus` extraction is an exact functional equivalence of the old `previewFor` helper. No regression.

### 4. `sendTriggers` threading

Acceptable scope for M6. The plan's Step 4 explicitly documents: "The `_sendTriggers` value is intentionally ignored for the drawer path in M6 (wiring it fully is M7 scope)." For the kanban path: `changeStatus(cardId, toStatus, orderVersion)` is the single-order M1 endpoint which has no `sendTriggers` parameter. The dialog accepts the toggle correctly (tests verify `onConfirm(true)` and `onConfirm(false)` both call `changeStatus` with the same arguments), and the UI correctly shows or hides the "Wyślij wiadomość" button based on trigger presence. Full sendTriggers wiring to the backend API is M7 scope for both paths. Not a bug.

### 5. Orphaned `StatusChangeConfirm.tsx`

Deleted in Stage 2 fixup. The file had zero imports remaining after Stage 1's extraction. Deleting now prevents dead code accumulating. Confirmed via `grep -r "StatusChangeConfirm"` — only the file itself appeared in results. Typecheck, lint, and tests all pass after deletion.

### 6. Closure capture in revert path

Not paranoia, but correctly handled. `onConfirm` closes over `snapshot` from its dependency array (`[pendingMove, snapshot]`). The `savedSnapshot = snapshot` capture is synchronous before the `await changeStatus(...)`. The only scenario where `snapshot` could differ between callback creation and execution is if another drag completes during the in-flight PATCH — but this is structurally impossible: while `pendingMove !== null`, the `StatusChangeTriggerDialog` is open (modal), which prevents dnd-kit from initiating a new drag. So the closure is safe in practice, and `savedSnapshot` will always match the pre-drag state. The test `"onConfirm reverts move and sets error toast on failure"` confirms the path works end to end.

### 7. Test coverage

All 6 plan-specified hook scenarios covered: same-column no-op, cross-column optimistic+dialog, cancel revert, confirm success, confirm with `sendTriggers=false`, confirm error revert. All 7 dialog scenarios covered: open=false renders nothing, status labels rendered, Wyślij button with match trigger, Tylko-zmień always present, `onConfirm(true)`, `onConfirm(false)`, `onCancel`. Double-drag while mutation in-flight is structurally prevented (modal blocks dnd-kit). Network abort is handled by the same catch path as 4xx — covered indirectly by the error revert test. Coverage is adequate for the plan scope.
