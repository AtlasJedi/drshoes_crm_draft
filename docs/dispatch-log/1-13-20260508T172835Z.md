## Decisions / deviations from plan

### LOC slightly exceeds 80 per file
- `ClientPicker.tsx`: 101 LOC (80 guideline). Already extracted `ClientPickerResults.tsx` (51 LOC) per plan instruction "If picker exceeds, extract ClientPickerResults.tsx". The remaining logic in ClientPicker is all essential: state declarations, debounce effect, select handler, conditional chip/input render.
- `ClientCreateModal.tsx`: 96 LOC (80 guideline). The form has 4 fields + error + 2 buttons — hard to split further without artificial fragmentation. Under 120 LOC cap; documented here.

### Keyboard navigation deferred
Keyboard arrow-up/down/Enter navigation was explicitly marked optional in the task spec. Skipped because:
- `ClientPicker.tsx` is already 101 LOC with the existing logic.
- Mouse-only interaction is sufficient for the admin panel use case.
- Deferred to a polish task in milestone 1 or later.

### `getClient(id)` fetch on selection
Per task spec: "fetch via getClient(id) on selection — slight extra request, much safer". Implemented — ensures downstream `onChange` always receives a full `ClientDto` (not the search result shape), which is required by task 1-15 new-order page.

### onBlur closes dropdown; onMouseDown uses preventDefault
Used `onMouseDown` with `e.preventDefault()` in `ClientPickerResults.tsx` so the click fires before the input's `onBlur` closes the list. This is the standard pattern for custom comboboxes.

### `@radix-ui/react-dialog` as runtime dep (not devDep)
Task spec said "fall back to runtime dep if version solving fails". Radix resolved cleanly as a runtime dep; placed it in `dependencies` (not `devDependencies`) since it ships in the client bundle.

### No `clsx` import — not installed
`grep -r clsx apps/web/` returned no matches. Used inline ternaries and direct class strings instead.
