## Decisions / deviations from plan

1. **New `MessageRouter.sendRetry()` method added (plan errata).** Plan note at line 4966
   flagged that `sendManual` would throw when `templateId=null`. Implemented
   `sendRetry(MessageEntity orig, UUID actorId)` that bypasses template lookup entirely
   and uses the stored `body`/`subject`/`channel` directly. `MessageRetryService.retry()`
   now calls `sendRetry` instead of `sendManual`. Unit test stubs updated accordingly.

2. **IT uses `JdbcTemplate.execute("DELETE FROM message")` in `@AfterEach`** instead of
   `messageRepository.deleteAll()`. Reason: `message.retry_of_message_id` is a
   self-referential FK (V010). After a successful retry, two rows exist with one referencing
   the other — JPA `deleteAll()` cannot determine deletion order and throws a
   `DataIntegrityViolationException`. Native SQL bypasses the constraint ordering issue.

3. **Plan IT template not used verbatim.** Plan's template:
   - Called non-existent `createClientAndReturnId(String, String, String)` and
     `createOrderAndReturnId(UUID)` helpers on `AdminWebTestBase`.
   - Tried reflection on `threadId` field to bypass the `message_thread_id_fkey` FK
     with a random UUID that was never persisted → FK violation at insert time.
   - Used bare `mockMvc` field instead of `mockMvc()` session-aware wrapper method.
   Real IT: creates client/order in `@BeforeEach`, creates a real `MessageThreadEntity`
   row in `buildMessage()`, uses `mockMvc()` wrapper throughout.

4. **Plan placed retry endpoint in `MessagesController` (not a new class).** Followed
   the plan's Step 3 — same controller, removed class-level `@RequestMapping`, made
   all paths explicit. The plan also suggested a new `MessageRetryController` in the
   dispatch prompt preamble, but Step 3 of the plan section itself says to add to the
   existing controller. Followed plan section (source of truth).

5. **`MessageRouter.sendRetry` LOC: ~70** (293 total for the whole file after addition).
   sendRetry is a duplicate of the private `send(...)` dispatch path minus template
   rendering. Could be refactored to share more code but kept separate for clarity
   per "simplest-but-working" directive.
