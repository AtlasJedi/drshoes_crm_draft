## 3. Three Flags Resolution

### Flag 1 — Radix substitution (hand-built dropdown)

**Decision: APPROVED with inline a11y fixups.**

Stage 1's no-Radix approach is correct (no new dep without Opus approval). However, three a11y gaps were found and fixed inline:

**Gaps found:**
- No Esc key handler — dropdown did not close on Escape
- No focus return to trigger after close — keyboard users stranded after selection
- No `aria-haspopup` / `aria-expanded` on trigger button
- Dropdown items had no `role="menu"` / `role="menuitem"` ARIA roles

**What is fine without change:**
- Outside-click handler: correctly registered only when `open===true`, correctly cleaned up on unmount — NO listener leak
- Tab keyboard reach: list items are `<button>` elements, Tab naturally reaches them — PASS
- Arrow-key navigation: not implemented, not required by plan; Tab-based access is sufficient for this component

**Fixes applied in `TemplatePicker.tsx`:**
1. Added `useCallback`-wrapped `close()` function that calls `setOpen(false)` + `triggerRef.current?.focus()`
2. `useEffect` for `open` now registers both `mousedown` (outside click) and `keydown` (Escape) handlers, both cleaned up in the same return
3. Trigger `IconBtn` now receives `triggerRef`, `aria-haspopup="menu"`, `aria-expanded={open}`
4. Dropdown `<div>` gains `role="menu"`; each `<button>` item gains `role="menuitem"`

**Fixes applied in `IconBtn.tsx`:**
- Added optional props: `triggerRef?: Ref<HTMLButtonElement>`, `aria-haspopup`, `aria-expanded`
- `ref={triggerRef}` forwarded to the underlying `<button>`
- Both ARIA attrs forwarded to button element

### Flag 2 — ReplyComposer at 142 LOC vs 80-LOC cap

**Decision: ACCEPT AS-IS.**

The 80-LOC cap is a "flagged for split" rule aimed at logic density, not JSX verbosity. Breakdown of ReplyComposer.tsx (142 lines including blank lines / comments):
- Lines 1–63: imports, interface, log setup, `handleSend`, `handleKeyDown`, derived variables — ~35 lines of actual logic
- Lines 64–142: JSX return — ~78 lines of layout, all of which are either multi-line Tailwind className strings or inline SVG path data

The hook extraction already removed all state management. No further split is warranted: extracting `ChannelToggle` or `ComposerToolbar` would create two stateless pass-through wrappers with no independent logic, adding indirection without reducing complexity. The 80-LOC cap's intent ("keep business logic graspable in one screen") is satisfied.

**Rationale recorded** per dispatch protocol rule #6 (flag, document, continue).

### Flag 3 — WHATSAPP channel edge

**Decision: DEFENSIVE DEFAULT implemented inline in `useReplyComposerState.ts`.**

If `thread.channel === "WHATSAPP"`, the original hook would initialise `channel` state to `"WHATSAPP"`, causing:
- Neither EMAIL nor SMS pill to be active-styled
- `handleSend` to POST `channel: "WHATSAPP"` — undefined behavior for M5's EMAIL/SMS-only reply API
- "wyślij na" hint showing the wrong address

Fix: added `resolveDefault(ch: Channel): Channel` helper that returns `"EMAIL"` for any channel that is not `"EMAIL"` or `"SMS"`. This is a 5-line pure function at module scope; it does not change the hook's public interface.

M5 scope guarantees only EMAIL/SMS threads reach the composer in practice. The defensive default ensures the component is in a well-defined state if that assumption is ever violated.
