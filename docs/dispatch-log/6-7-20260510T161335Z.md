## Decisions

1. **Timezone for "today"**: `Europe/Warsaw` used both in tests (for seeding) and in the controller (for converting `LocalDate` params to `Instant` window boundaries). `to` date is inclusive: controller converts `to.plusDays(1).atStartOfDay(WARSAW)` so the full last day is covered.

2. **Range guard**: `ChronoUnit.DAYS.between(from, to) > 92` → 400. Exactly 92 days is allowed; 93+ is rejected.

3. **"No due date" handling**: Orders with `planned_pickup_at IS NULL` go into the `unscheduled` list, capped at 50 by `received_at DESC`. They always appear regardless of the `from`/`to` window — this matches the plan's intent (the calendar sidebar shows the full unscheduled backlog).

4. **Paging**: `findUnscheduled()` uses `LIMIT 50` in native SQL (no Spring `Pageable`) — keeps it simple and consistent with plan spec.

5. **Count semantics for `isUrgent`**: Replicated M1 derivation exactly — tag `"pilne"` in JSONB array OR `plannedPickupAt` within 48 hours of now.

6. **Batch fetching**: Client names and item summaries are fetched in two batches (one `findAllById` + one loop over `findAllByOrderIdOrderByPosition` per order). No N+1 on client names; item fetch is per-order (no bulk `findAllByOrderIdIn` exists — acceptable at calendar scale).

7. **Controller LOC**: 118 LOC — within the 120 LOC hard cap per CLAUDE.md granularity rule.

8. **No migration needed**: Read-only endpoint querying existing columns (`planned_pickup_at`, `status`, `deleted_at`). `planned_pickup_at` already has an index from M1 (V004+). No V014 required.

9. **Duplicate `@RequestMapping` base**: `CalendarController` and `OrderController` both map to `/api/admin/orders`. Spring correctly disambiguates by method path (`/calendar` GET vs `/` GET and `/{id}` GET). No conflict.
