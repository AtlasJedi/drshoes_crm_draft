# Task 3-6-fixup — Order ownership guard + drop @Transactional from stream()

Two security/correctness fixes identified in stage-2 code-quality review of commit `ca521b6`.

## Files changed
- `backend/app/src/main/java/com/drshoes/app/photo/service/PhotoService.java` (modified)
- `backend/app/src/test/java/com/drshoes/app/photo/service/PhotoServiceTest.java` (modified — 3 new tests)

## Changes made

### Fix 1 — Cross-order photo ID enumeration guard (Issue 1)

`relabel`, `delete`, and `stream` all previously looked up a photo by `photoId` only, without verifying the photo belonged to the `orderId` supplied in the URL path. A user with access to one order could craft `/api/admin/orders/{otherOrderId}/photos/{photoId}` to read or mutate photos from a different order.

**Changes:**
- Added `orderId` parameter to `relabel(UUID orderId, UUID photoId, ...)`, `delete(UUID orderId, UUID photoId, ...)`, and `stream(UUID orderId, UUID photoId)`.
- Added private `verifyOwnership(UUID orderId, UUID photoId, Photo photo)` helper: throws `PhotoNotFoundException(photoId)` (not a more informative exception) on mismatch — leaks less to an attacker who cannot distinguish "not found" from "wrong order".
- `verifyOwnership` is called in all three methods immediately after the `findById` lookup.
- `listForOrder` was already safe — it queries `WHERE order_id = ?` so no fix needed.
- `upload` was already safe — it validates `orderId` via `validateOrderExists` and sets `photo.orderId = orderId` directly.

### Fix 2 — Drop @Transactional(readOnly=true) from stream() (Issue 2)

`stream()` was annotated `@Transactional(readOnly=true)` but returns a live AWS SDK socket-backed `InputStream`. The JPA transaction would remain open for the entire duration of the HTTP response body write, holding a JDBC connection pool slot for potentially minutes on slow clients.

**Change:** Removed `@Transactional` from `stream()`. Spring Data's `findById` opens its own short transaction internally. The JDBC connection is released before the blob `InputStream` is handed to the response writer. Added a one-line comment in the Javadoc explaining the rationale.
