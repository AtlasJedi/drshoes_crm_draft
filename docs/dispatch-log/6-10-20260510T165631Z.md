## Decisions

1. **Vitest setup (inline prerequisite):** The plan's Step 2 ("Expected: FAIL — Cannot find module './api-server'") and Step 5 ("PASS — 5 tests passing") both require a working vitest runner. `apps/web` had only a `echo "(playwright in milestone 1)"` placeholder. Vitest (`v4.1.5`), `@vitejs/plugin-react`, and `vite-tsconfig-paths` were added as devDeps; `vitest.config.ts` created with `environment: node`. The test script was updated to `vitest run`. This is documented as an inline prerequisite; no prior tests existed in `apps/web` so there is no regression risk.

2. **`log.error` in `serverGet`:** Plan specifies `log.error` on non-ok responses. `lib/messaging/api-server.ts` uses `log.warn`. Chose `log.error` per plan spec — HTTP error responses from the backend are unexpected at the server-fetch layer and warrant ERROR severity.

3. **Non-null assertions in test (`[0]!`):** `tsconfig` strict mode flags `array[index]` as `T | undefined`. Used `[0]!` non-null assertions in the test file since the test payload is controlled fixture data and null safety is not the concern under test. This keeps the plan's test code intact while satisfying `tsc --noEmit`.

4. **`MixByTypeRowDto` TS comment updated:** Docstring says "Mirrors `DashboardChartsDto.OrdersPerWeekRowDto`" — corrected to `DashboardChartsDto.MixByTypeRowDto` in the implementation (plan's comment was for `MixByTypeRowDto` but said `OrdersPerWeekRowDto` — plain copy-paste artifact in the plan spec).

5. **`kind` as union vs `string`:** Java sends `kind` as the enum's `.name()` — always one of `NAPRAWA`, `CUSTOM_BUTY`, `CUSTOM_KURTKA`. Using a TS union literal type provides compile-time safety and satisfies the ERRATA rule that TS unions must use the Polish names.

6. **No calendar / kanban fetchers added:** This task is scoped to KPI + chart endpoints only (6-5 and 6-6 backends). Calendar (6-7) and Kanban (6-8) fetchers will be created in their respective frontend tasks (6-13, 6-17+).
