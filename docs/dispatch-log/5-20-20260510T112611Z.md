## Decisions

1. **`effectiveChannel` auto-fallback behavior**: When the user selects `channel = "EMAIL"` but the chosen client has no email (`!hasEmail`) and has a phone (`hasPhone`), `effectiveChannel` auto-promotes to `"SMS"`. The channel toggle visual state: the EMAIL button renders with `effectiveChannel === ch` check, so it shows as inactive (not highlighted) and the SMS button shows as active. This correctly reflects what will actually be sent. The EMAIL button also has `disabled={!hasEmail}` so the user cannot manually re-select EMAIL for this client — they must either clear the client and pick one with email, or accept SMS. This matches the plan's intent for the auto-fallback.

2. **`NewMessageDialog` mounted after flex-row `</div>`**: Placed inside the root `<div className="flex flex-col h-screen bg-paper">` but after the `<div className="flex flex-1 overflow-hidden">` layout container. Radix `Dialog.Portal` renders via React portal to `document.body` regardless of DOM position, so placement is purely for code organization — putting it at the end of the component tree is the conventional pattern (consistent with how Radix dialogs are mounted in other components in this codebase).

3. **`subject` passed as `undefined` for SMS**: `subject: effectiveChannel === "EMAIL" ? subject : undefined` — this matches `SendNewRequest.subject?: string | null` (optional field). For SMS, subject is omitted entirely from the request body.
