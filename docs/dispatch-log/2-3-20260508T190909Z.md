## Decisions / deviations from plan

**`scheduled_message_id` handling:** Omitted entirely from `MessageEntity`. V001 adds it via a deferred `ALTER TABLE` with FK to `scheduled_message`. The plan recommendation was to omit it in M2 (no service writes it), and that is what was done. Will be added in M3 when `ScheduledMessageEntity` is introduced.

**`createdAt` / `updatedAt` on `MessageTemplateEntity` and `TriggerEntity`:** Both columns are `insertable=false, updatable=false`. The DB trigger (`touch_updated_at()`) manages `updated_at`; Hibernate must not try to write these columns. This differs slightly from `Order.java` which uses `Instant.now()` defaults with `@PreUpdate` — but for messaging entities the DB-trigger pattern is more correct given V001's explicit `TIMESTAMPTZ NOT NULL DEFAULT now()` + trigger.

**`updatedAt` on `MessageEntity`:** Not mapped — the `message` table has no `updated_at` column (append-only per V001 schema). Only `created_at` is present.

**`channel` in `MessageTemplateEntity` / `MessageEntity`:** Stored as raw `String` (not the microlib `Channel` enum). ERRATA item 12 says to use the microlib enum where natural; however, the DB schema uses CHECK constraints (`VARCHAR(16) CHECK (channel IN ('EMAIL','SMS','WHATSAPP'))`), and mapping a raw String avoids any JPA column-type mismatch. The microlib `Channel` enum is available for use at the service layer when converting to/from the entity. No domain-side mirror needed.

**`eventParams` and `channels` in `TriggerEntity`:** Stored as `String` with `columnDefinition = "jsonb"`. Deserialization is deferred to the service layer. This matches the plan body exactly.

**`TriggerEntity.enabled`:** Using `Boolean` (boxed) to match the plan body and nullable-safe patterns; `TriggerRepository.findAllByEventAndEnabledTrue` uses Spring Data's `enabledTrue` convention which correctly queries `enabled = true`.

**`Instant` vs `OffsetDateTime`:** `Order.java` uses `Instant`; the plan body for messaging entities specifies `OffsetDateTime`. Used `OffsetDateTime` throughout messaging entities as directed by the task spec and ERRATA item (OffsetDateTime matches DB `TIMESTAMPTZ` more naturally in Spring Data projections).
