## Decisions

1. **Table name correction:** Plan's IT used `INSERT INTO orders` — actual table is `order_`. Corrected in `OrderListExtendedIntegrationTest.java`.

2. **tags NOT NULL:** `tags` column has `NOT NULL DEFAULT '[]'::jsonb`. orderC in IT uses `'[]'::jsonb` (not NULL as in plan draft).

3. **AdminWebTestBase + MockMvc:** Plan spec used `TestRestTemplate` from `AbstractIntegrationTest` — this doesn't handle Spring Security session auth and returns 401. Switched to `AdminWebTestBase` (MockMvc + `loginAsOwner()`) per existing pattern in `OrderControllerIntegrationTest`.

4. **jsonb_contains instead of jsonb_array_contains_text:** Plan noted `jsonb_array_contains_text` as option with fallback. `jsonb_array_contains_text` is not a real Postgres function. Used `cb.function("jsonb_contains", Boolean.class, root.get("tags"), cb.function("jsonb_build_array", Object.class, cb.literal(tag)))` — `jsonb_contains` IS the real Postgres function backing `@>`. No custom Hibernate dialect registration needed.

5. **api.ts also needed fixing:** The client-side `lib/orders/api.ts` `buildOrdersQuery` also called `params.set("status", filters.status)` — this broke typecheck after `status` was widened to `OrderStatus | OrderStatus[]`. Extended it with the same array-handling logic as api-server.ts.

6. **filtersInitial.status:** The `OrdersFilters` component expects a single `OrderStatus` (not array). Used `status?.[0]` to pass the first status value — preserves single-filter display compat while multi-status arrives via URL.

7. **Vitest warning about nested vi.mock:** Plan test case 5 ("active chip has visual highlight") calls `vi.mock` inside an `it()` body. Vitest hoists it automatically and warns. This is from the plan spec verbatim and doesn't cause failures. Left as-is to match spec; noted as follow-up to clean up.
