## decisions

### 1. BlobStorage interface mismatch ŌĆö extensive adaptation

The plan's test code and implementation skeleton assumed a different API than what exists in the codebase:

| Aspect | Plan assumed | Reality |
|--------|-------------|---------|
| `BlobKey` constructor | `new BlobKey("photos", "2026/05/abc.jpg")` (2-arg: bucket + path) | `new BlobKey("photos/2026/05/abc.jpg")` (1-arg: full value) |
| `BlobStorage.put` return type | `BlobMetadata` | `void` |
| `BlobStorage.put` content arg | `byte[]` | `InputStream` |
| `BlobStorage.get` return type | `byte[]` | `InputStream` |
| `BlobMetadata` fields | `(BlobKey key, int size, String contentType, Instant now)` | `(String contentType, Long contentLength)` |
| `BlobStorage.exists` | Not in plan | Present in interface ŌĆö implemented |
| `BlobStorage.presignPut` | Not in plan | Present in interface ŌĆö implemented |

All tests and implementation adapted to match the real interface. The presigned URL convention (`/api/admin/photos/local/{key.value()}`) is preserved as specified.

### 2. Path-traversal: "photos/../escaped.jpg" is NOT a traversal

The plan's original test used `new BlobKey("photos", "../escaped.jpg")` (non-existent 2-arg constructor) testing that `photos/../escaped.jpg` is blocked. Investigation reveals: `root.resolve("photos/../escaped.jpg").normalize()` = `{root}/escaped.jpg` ŌĆö which is **inside root**. This is NOT a traversal. The dangerous form is a **leading `..`** that climbs above root: `../escaped.jpg` ŌåÆ `{root.parent}/escaped.jpg`.

Tests 7, 8, 9 use actual traversal paths (`../escaped.jpg`, `photos/../../etc/passwd`) that genuinely escape root after normalization.

### 3. Root normalization at construction ŌĆö the critical correctness fix

`this.root = root.toAbsolutePath().normalize()` is done at construction, not at each `resolve()` call. This means `resolved.startsWith(root)` compares two consistently normalized paths. If root were stored non-normalized (e.g., with a trailing `/` or a `//`), `startsWith` could produce false negatives on some JVM/OS combinations. The plan's implementation code had this correct; retained.

### 4. presignGet/presignPut call resolve() for traversal check

Both presign methods call `resolve(key)` even though they only build a URL string. This ensures an attacker cannot use presign to probe the existence of paths outside root (the URL would be returned without any filesystem access, but the key is still validated).

### 5. Null-byte test expectation is RuntimeException (not IAE)

The null-byte test (`put_rejects_null_byte_in_key`) asserts `RuntimeException.class` (not `IllegalArgumentException`) because `BlobKey` does not validate for null bytes ŌĆö the first failure point is `root.resolve(key.value())` which on macOS/Linux throws `InvalidPathException extends IllegalArgumentException extends RuntimeException`. The test uses the broader supertype to remain portable.

### 6. Percent-encoded segments test documents safe passthrough

`"photos/%2e%2e/encoded.jpg"` resolves to `{root}/photos/%2e%2e/encoded.jpg` ŌĆö the literal `%2e%2e` directory name, not `..`. This test documents that `LocalFsBlobStorage` does NOT decode percent-encoding (URL decoding belongs at the HTTP/servlet layer). The test asserts the file is created at the literal path (safe) and no exception is thrown.
