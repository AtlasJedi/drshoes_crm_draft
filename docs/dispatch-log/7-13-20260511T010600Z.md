## Decisions / Surprises

1. **`listOrdersServer` alias confirmed:** `lib/clients/api-server.ts` exports `listOrdersServer(opts: {clientId, page?, size?})` — distinct from `lib/orders/api-server.ts`'s `listOrdersServer(filters, page, size)`. Imported as `listClientOrders` per plan alias instruction. No conflict.

2. **`getClientServer` double-fetch acceptable:** The layout (`layout.tsx`) also calls `getClientServer` for the 404 guard. The page makes a second call to get the `ClientDto` for `<ClientHeader>`. This is consistent with the 7-12 pattern and matches the plan's code block exactly. Minor double-fetch, no functional issue.

3. **`ClientHeader` import path:** `../\_components/ClientHeader` — one level up from `zlecenia/` into `[id]/`. Confirmed correct by checking directory structure.

4. **No `Page` import needed for response type:** The `ordersPage` type is inferred from `listClientOrders` return type (`Promise<Page<OrderListRow>>`). Destructured directly.

5. **Duplication flag (hygiene carry-forward):** `lib/clients/api-server.ts::listOrdersServer` and `lib/orders/api-server.ts::listOrdersServer` both hit `GET /api/admin/orders` but with different parameter shapes. The clients variant only passes `clientId`+`page`+`size`; the orders variant has the full `OrderListFilters` object (status, type, q, etc.) without `clientId`. Duplication is justified by the different caller shapes, but `OrderListFilters` could gain an optional `clientId` field in a future hygiene milestone, allowing the clients module to delegate to the orders module instead.
