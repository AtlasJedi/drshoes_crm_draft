## Why

Plan §5-20's `NewMessageDialog.handleSend` calls `onSent(result.threadId)` against the return of `sendNewToClient(...)`. That call returns `Promise<MessageDto>`, but `MessageDto` did not carry a `threadId` field — neither in the Java record nor the TS interface — so the plan code would not compile.

Resolution per "simplest-but-working over engineering ceremony": add `threadId` to `MessageDto`. The data is already on `MessageEntity` (`getThreadId()` exists, line 123 in `MessageEntity.java`); only the DTO and its 5 mapper sites needed updating. No backend behavioral change. No test contract churn (`grep`'d — no test JSON-asserts on MessageDto fields; the only `$.threadId` assertions are on `UnreadElsewhereDto`).

Done inline rather than as a subagent because the diff is ~20 LOC across 6 files, mechanical (positional record arg + 5 mappers each appending `m.getThreadId()`), and zero risk.

Field added at the END of the record (position 18) so positional constructors only need a new arg appended — minimal diff. Marked non-nullable: every Message belongs to exactly one Thread per the M5 schema (V012 + earlier).
