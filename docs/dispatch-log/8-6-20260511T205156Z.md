## Decisions

1. **`RecipientHashUtil.hashFirst8Hex()` reused, not duplicated.** The plan's `MessagingSpanHelper` code block includes a private `recipientHash()` method that duplicates the logic already in `messaging-core`. Carry-forward directive ("USE THIS — do not duplicate") was honoured: `MessagingSpanHelper` imports and delegates to `RecipientHashUtil`. This keeps the lib as the single source of truth for the hashing contract.

2. **`IllegalArgumentException` propagation through span.** When `Channel.valueOf("PIGEON")` throws inside the lambda, `dispatchWithSpan` catches it as `Exception`, sets span status `ERROR`, and rethrows. The existing test `dispatch_unknownChannelString_throwsIllegalArgument` still passes because the IAE propagates out of `dispatch()` as before. This is consistent with the plan intent.

3. **Gateway `RuntimeException` caught by dispatcher — span status stays `OK`.** When an email gateway throws (e.g. `SMTP timeout`), the lambda's `catch (Exception e)` clause catches it, marks the message `FAILED`, and returns `persisted` normally. From `dispatchWithSpan`'s perspective, the supplier returned successfully → span gets `StatusCode.OK`, not `ERROR`. This is the correct behaviour (the span tracks "did the dispatch method complete" not "did the send succeed"). The new `dispatch_gatewayFailure_emitsSpan` test only asserts span name, not status — matching the plan's comment at lines 1094–1097. Stage 2 may want to review whether `OK` is the right status here or if the gateway failure should propagate differently.

4. **LOC budget.** `MessageGatewayDispatcher` went 105 → 111 LOC (still under 120 — the `dispatchWithSpan` wrapper adds only 2 lines net). `MessagingSpanHelper` is 63 LOC. Both within budget. The LOC target comment in the dispatcher javadoc says "≤ 90" (old pre-8-6 target) — not updated here; Stage 2 may flag.

5. **Spring bean injection.** `MessageGatewayDispatcher` remains `@Service`. Spring Boot OTel starter 2.10.0 (on classpath since 8-4) registers an `OpenTelemetry` bean automatically. The new `OpenTelemetry otel` constructor parameter will be injected without any `@Bean` or config change needed.
