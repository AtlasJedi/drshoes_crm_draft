## Decisions / deviations from plan

**JdbcTemplate instead of TriggerFireRepository.saveAndFlush():**
The plan body uses `fires.saveAndFlush(new TriggerFireEntity(...))`. This fails silently on the
second call: Hibernate's `SimpleJpaRepository.save()` uses `isNew()` detection based on whether
the entity has a null ID. For `@IdClass` composite-PK entities with manually-assigned IDs, all
IDs are set — Hibernate treats the entity as "existing" and issues a SELECT + no-op UPDATE rather
than an INSERT, bypassing the DB unique constraint entirely.

Switched to `EntityManager.persist()` + `em.flush()` as a first fix (always INSERTs). That also
failed because with `REQUIRES_NEW`, each call gets a fresh transaction-scoped persistence context,
but the second `em.persist()` on the same logical key in a new context still doesn't guarantee
a flush-time constraint hit in all cases.

Final fix: `JdbcTemplate.update()` with a literal `INSERT INTO trigger_fire ...` SQL. This
bypasses the JPA cache entirely and always issues a real DB INSERT. On PK collision Postgres raises
`unique_violation`; Spring wraps it as `DuplicateKeyException` (a `DataIntegrityViolationException`
subtype), which is caught and logged as `outcome=duplicate`. This is the most reliable approach
for idempotency guards and is consistent with the plan's intent.

**Order-creation helper — local JDBC in test (not base class):**
`AbstractIntegrationTest` has no order-creation helper. Per the task instructions, added a local
`createMinimalOrder()` method inside the test class itself (not in the base class). It uses
`JdbcTemplate` to INSERT a `client` row (satisfying the `client_contact_present CHECK` with
`phone='+48 600 000 000'`) and an `order_` row (`id`, `code`, `client_id`,
`status='WSTEPNIE_PRZYJETE'`, `version=0`). An `@AfterEach` cleanup deletes `trigger_fire` rows
first (FK), then `order_`, then `client` to avoid constraint violations between tests.

**`DuplicateKeyException` vs `DataIntegrityViolationException`:**
Caught `DuplicateKeyException` specifically (a subclass of `DataIntegrityViolationException`)
since we know the only possible violation here is a PK duplicate — this avoids masking real
constraint errors (FK violations, NOT NULL violations) that would indicate a bug.
