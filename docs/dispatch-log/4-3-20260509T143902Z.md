## Decisions / deviations from plan

1. **Attachment record mismatch (plan errata)**
   Plan test code used `new Attachment("report.pdf", "application/pdf")` (2-arg, fields `name` + `contentType`). The actual `Attachment` record in messaging-core has signature `(String storageKey, String mime, Long bytes)`. Tests adapted to use `new Attachment("report.pdf", "application/pdf", (long) pdfBytes.length)`. Mapper uses `att.storageKey()` and `att.mime()` instead of the plan's `att.name()` / `att.contentType()`.

2. **Jackson-databind not in email-gateway pom**
   `PostmarkResponseMapper` requires `ObjectMapper`. email-gateway pom previously had no Jackson dependency (only slf4j + spring-boot-autoconfigure). Added `jackson-databind` without version — covered by `spring-boot-dependencies` BOM already imported in parent pom. No version conflict risk.

3. **4 ResponseMapper tests instead of 2**
   Plan noted "2 explicit ResponseMapper cases; the 2 pattern-noted cases are not compiled at this step." Implemented all 4 cases (200+EC0, 200+EC≠0, 422, 500) in the test class since the plan text described them fully. Raises total from plan's expected 6 to 8 tests. All green.

4. **Body-sniff rule applied as specified**
   `body.contains("<")` is used (not `startsWith`). Plan errata #1 noted in class Javadoc: callers with `<`-containing non-HTML bodies must HTML-encode or await a future flag.

## Concerns
- Attachment `storageKey` is used as the Postmark `Name` field. For display in email clients this will show storage keys (e.g. UUIDs) not human-friendly filenames. The 4-4 gateway task should consider mapping to a display name; the current design matches the plan body (att.name → Postmark Name) but with the renamed field. Flag to 4-4 author.
