## Decisions / deviations from plan

1. **spring-security-test missing from pom**: `CsrfEnforcementTest` imports from `spring-security-test`
   but the dependency was not in `pom.xml`. Added `spring-security-test` test-scope. This was a
   latent omission — the CsrfEnforcementTest only uses the bean, not `SecurityMockMvcRequestPostProcessors`,
   so it compiled without it before; `ClientControllerIntegrationTest` needs `csrf()` directly.

2. **AdminWebTestBase uses SecurityMockMvcRequestPostProcessors.user() not session injection**:
   Plan sketched `loginAsOwner()` storing a MockHttpSession with a SecurityContextImpl.
   Spring Security 6's `DeferredSecurityContext` / `SecurityContextHolderFilter` chain does not
   load context from a raw `MockHttpSession` attribute the same way the old
   `HttpSessionSecurityContextRepository` did in Spring Security 5. Using
   `SecurityMockMvcRequestPostProcessors.user(...).roles(...)` is the idiomatic SS6 approach
   and produces correct 403/401 differentiation.

3. **Audit aspect does NOT cover ClientController**: The `AuditLogAspect` pointcut is
   `com.drshoes.app..api..*Controller` — `ClientController` lives at
   `com.drshoes.app.client.ClientController` (no `.api.` segment) and is therefore not audited
   by the aspect. The plan says "verify by reading the aspect". Verified: no coverage.
   Leaving aspect unchanged to avoid regression in existing audit tests. Controller has
   its own structured INFO logging per dispatch-protocol §7 on every endpoint.

4. **RbacService bean name is `@Component("rbac")`**: `@PreAuthorize` expression uses
   `@rbac.canManageClients(authentication)` (not `@rbacService`). Verified in RbacService source.
