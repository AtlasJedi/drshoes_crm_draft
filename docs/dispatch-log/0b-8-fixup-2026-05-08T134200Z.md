# Task 0b-8-fixup — Exclude @ExceptionHandler methods from audit pointcut

## Files changed

- `backend/app/src/main/java/com/drshoes/app/audit/AuditLogAspect.java` (modified, +9 / -3):
  - Pointcut now: `execution(public * com.drshoes.app..api..*Controller.*(..)) && !@annotation(org.springframework.web.bind.annotation.ExceptionHandler)`.
  - Removed unused imports `org.springframework.transaction.annotation.Propagation` and `Transactional` (the `@Transactional(REQUIRES_NEW)` lives correctly on `AuditLogWriter.write()`, not on the aspect).
  - Added a comment block explaining why exception-handler methods are excluded.
- `backend/app/src/test/java/com/drshoes/app/audit/AuditLogAspectIntegrationTest.java` (modified, +14 / -1):
  - Strengthened existing `login_writes_audit_log_row` assertion from `>= 1` to `== 1` to lock single-row behaviour.

## Decisions / deviations

- **Chose pointcut exclusion over try/catch refactor.** Reviewer offered two fix paths: (a) `&& !@annotation(ExceptionHandler)` exclusion, (b) catch the auth exceptions explicitly in the aspect and map to the correct status. Path (a) is the simpler, mechanical change; path (b) couples the aspect to specific exception types and re-implements logic Spring MVC already does. Picked (a).
- **Strengthened existing assertion** from `>= 1` to `== 1`. Originally the test was permissive enough to pass with the bug; tightening to exactly-one closes the door on regression even on the success path.
- **Did not address the Minor `actorId` always-null issue** — that is documented intentional debt; persisting actor email-string would change the entity schema (new column) and is out of scope for this fixup.
- **Did not address the Minor dead-code note on `InetAddressConverter`** — read-path defensive scaffolding is fine; flagged the `convertToDatabaseColumn` half as a TODO comment is a noise change and was deferred.
