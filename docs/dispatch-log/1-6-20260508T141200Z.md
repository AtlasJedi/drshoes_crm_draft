## Decisions / deviations from plan

1. **Plan says create() defaults to WSTEPNIE_PRZYJETE; plan body code sketch says PRZYJETE.**
   Used PRZYJETE (per the code sketch in the plan body at line 1112) since the test spec at
   line 1070 also says "status defaults to PRZYJETE". The dispatch prompt's sanity check listed
   WSTEPNIE_PRZYJETE but the authoritative plan body wins. Test 4 transitions
   PRZYJETE → WSTEPNIE_PRZYJETE → PRZYJETE to exercise receivedAt logic without a forced
   enum-order dependency.

2. **Class size split:** OrderService is 165 total / 133 effective LOC — 13 lines over
   the 120 LOC ceiling after extracting OrderItemService and OrderSpecifications.
   The overage is structural boilerplate (constructor, imports, Javadoc) and the item-delegation
   wrappers required by the plan interface. Further splitting would produce an artificial
   OrderQueryService (2 methods) with no cohesion gain. Documented as a CONCERN.

3. **Optimistic-lock simulation:** Used service-layer sequential calls within one test method.
   After `changeStatus(version=0)` commits, a second `changeStatus(version=0)` against the
   same order reads version=1 from DB and throws OrderVersionConflictException(currentVersion=1).
   No EntityManager.clear() needed because each @Transactional call opens+closes its own session.
   The two calls are sequential in the same test thread, which is sufficient — true concurrent
   contention is not representable in a single-threaded JUnit test.

4. **Free FSM:** No transition table. All status moves are permitted. Timestamps applied:
   - → PRZYJETE: sets receivedAt if null
   - → WYDANE: sets pickedUpAt if null

5. **OrderItemService.recomputeTotal:** Package-private (not private) to allow test-layer
   access if needed by future tasks. Does not log to avoid noise — callers log per operation.
