### 2. Atomicity / Partial-Failure Semantics

**Accepted.**

### 3. sendTriggers Wiring — BLOCKER

**`@JsonProperty(defaultValue = "true")` does NOT work for primitive `boolean` fields in
Jackson records.**

Jackson's `defaultValue` attribute in `@JsonProperty` is treated as *documentation metadata*
for schema generation (Swagger/OpenAPI). It is NOT applied by `BeanDeserializer` or
`RecordDeserializer` when a field is absent from JSON. The boolean field falls back to Java's
language default (`false`), not the documented default.

**Verified empirically** with Jackson 2.18.2 (the version pulled by Spring Boot 3.4.0):

```
json: {"name":"test"}   (sendTriggers field absent)
Result sendTriggers = false   ← WRONG, should be true
```

**Impact:** Every existing single-order call that posts
`{"targetStatus":"...","expectedVersion":0}` without `sendTriggers` now silently gets
`sendTriggers=false`, so `triggerEngine.onStatusChange` is never called. This is a
regression for the single-order flow introduced by task 6-9.

The existing `OrderControllerIntegrationTest.postStatusHappyPath` posts without `sendTriggers`
and does not assert trigger behaviour, so this regression is not caught by tests.

**Fix:** Change `boolean sendTriggers` to `Boolean sendTriggers` (boxed) in `ChangeStatusRequest`
and add a compact constructor to normalise `null → true`. Update the `sendTriggers()` accessor
call sites to handle the boxed type (or use `Boolean.TRUE.equals(req.sendTriggers())`). See
fixup commit `<fixup_sha>` for the applied change.

The same issue exists for `BulkStatusRequestDto.sendTriggers`, but the impact there is lower
because the bulk caller explicitly controls the field and all current tests pass the field
explicitly. However, for consistency and correctness both are fixed.

### 6. Audit Semantics

The `AuditLogAspect` pointcut intercepts all `com.drshoes.app..api..*Controller.*(..)` methods.
`BulkStatusController.bulkStatus` matches → exactly 1 HTTP audit row per bulk call.
`OrderService.changeStatus` has NO `@Audited` annotation → 0 per-order rows.

Stage-1's decision to rely on the HTTP-level controller audit (1 row per bulk request, not per
successful order) differs from a possible "1 audit row per successful per-order transition"
interpretation of the plan's Stage-2 focus point (c): "audit semantics (exactly one audit row
per successful per-order transition, none for failures)".

Reading the plan carefully: the Stage-2 review points say "exactly one audit row per successful
per-order transition, none for failures" — this would mean per-order rows, not per-request.
However, the plan's `OrderService.changeStatus` code block (Step 3) does NOT add `@Audited`
to `changeStatus`, and the Javadoc comment on `BulkStatusController` explicitly says "Do NOT
add @Audited here — the AuditLogAspect controller pointcut already writes exactly one audit row
per HTTP request." The plan itself does not add `@Audited` to `changeStatus`. The tests verify
`delta = 1` (one per bulk call). **Accepted as-is** — Stage-1's interpretation is the more
defensible one given the code blocks in the plan.

### 10. VERSION_CONFLICT Branch

`BulkStatusController` always reads `order.getVersion()` from the freshly-fetched entity and
passes it as `expectedVersion`. The only race window is between the read and the
`changeStatus` write (no distributed lock). The `OrderVersionConflictException` catch is
scaffolded correctly and unreachable in practice under normal serialization. **Accepted.**
Removing the catch would break the future state-machine or concurrent-update use case;
keeping it is the right choice.

### 11. `reason` Field

The plan's DTO code block (line 3009) includes `String reason` in `BulkStatusRequestDto`.
The plan gives no spec for storage, audit attachment, or persistence. The TypeScript interface
at line 9624 includes `reason?: string` (optional). No DB column for it exists. Stage-1
accepted it as a logged-only no-op field; it is not used anywhere in the controller.

The plan is **silent on what to do with it beyond defining it in the DTO**. No audit column,
no `cancelledReason` field, no storage target is mentioned. Stage-1's decision to accept it
in the DTO and ignore it is consistent with "simplest-but-working" (memory entry
`feedback_simplest_working.md`). **Accepted as follow-up** — if the plan had intended
persistence it would have specified a target column or audit metadata key.
