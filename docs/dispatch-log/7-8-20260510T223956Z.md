## Decisions / Surprises

1. **Plan had wrong import path in test (`../../../page`):** The plan specified `import("../../../page")` in the test, but the test lives at `.../clients/_components/__tests__/ClientsPage.test.tsx`. Three levels up from `__tests__/` reaches `admin/page.tsx` (the dashboard), not `clients/page.tsx`. Corrected to `import("../../page")` (two levels up from `__tests__/`). This is a straightforward path arithmetic fix — the intended target was always `clients/page.tsx`. Documented as a plan errata.

2. **`ClientSearchResultsTable` uses `ClientSearchResult` type directly:** The plan's reference code used an inline `interface SearchResult { id; fullName; phone; email }`. Since `ClientSearchResult` in `@/lib/clients/types` has exactly those four fields, I imported and used the canonical type instead of redeclaring. Cleaner, no behavior change.

3. **No `next/link` mock needed in test:** The tests only call `Page()` and render its output for `listClientsServer` (which returns paginated data rendered by `ClientListTable`). `ClientListTable` uses `next/link` internally but the test doesn't assert on link behavior — only on text presence. The mock is therefore not required (and `ClientListTable.tsx` is a separate module, not inlined in the page).

4. **No `lib/log.ts` mock needed:** The named logger is called with `log.info(...)` in the page, but `createLogger` returns a plain object with `info`/`warn`/`error` methods that write to stdout. No jsdom issue — the tests produce log output to stdout but do not fail because of it.

5. **`next/navigation` mock covers `ClientListSearchBox`:** The page renders `<ClientListSearchBox initialQ={q} />` which is a CC that calls `useRouter`. The `vi.mock("next/navigation", ...)` in the test file covers this correctly.
