## Decisions

1. **UpdateClientRequest arity — 7 fields, not 8.**
   Plan showed `(null, null, null, null, channel, null, rodoConsent, null)` — 8 args. Actual record
   in codebase has 7 fields: `(firstName, lastName, phone, email, preferredChannel, rodoConsent, notes)`.
   Corrected call: `new UpdateClientRequest(null, null, null, null, channel, rodoConsent ? Boolean.TRUE : null, null)`.

2. **CreateOrderItemRequest arity — 4 fields, not 5.**
   Plan showed `new CreateOrderItemRequest(kind, description, null, 0, 0)` — 5 args. Actual record:
   `(kind, description, craftsmanNotes, priceCents)` — 4 fields. Corrected to `(kind, description, null, 0)`.

3. **ChangeStatusRequest.expectedVersion is `int`, not `long`.**
   Plan used `long v = version`. Corrected to `int v = version` matching the record field type.

4. **ClientDto.id() is UUID — no String roundtrip needed.**
   Plan's factory used `ClientRecord(String id)` with `UUID.fromString(dto.id())`. Since `ClientDto.id()`
   is already `UUID`, simplified to returning `UUID` directly from `create()` and calling
   `clientRepository.findById(id).orElseThrow()`.

5. **WSTEPNIE_PRZYJETE transition from PRZYJETE.**
   `OrderService.changeStatus` has no state-machine guard — it accepts any `targetStatus` directly.
   Therefore `PRZYJETE → WSTEPNIE_PRZYJETE` works via a single `changeStatus` call with
   `sendTriggers=FALSE`. No JPA bypass needed. Documented in class Javadoc.

6. **sendTriggers=FALSE for all seed status changes.**
   Using `Boolean.FALSE` to suppress SMS/email triggers during seed. Avoids firing real triggers
   against non-existent phone/email recipients in the demo environment. Consistent with spec intent.

7. **Photo seed deferred.**
   `PhotoService.upload()` requires a live `BlobStorage` connection (MinIO/R2). In the Testcontainers
   context, there is no MinIO container wired. Photos are explicitly omitted from this task per plan
   step 3 and the errata note. Tracked as post-M8 hygiene: add MinIO Testcontainers + photo seed.

8. **DemoSeedRunner temporary guard retained.**
   The 8-8 skeleton added `if (!clients.isEmpty() && !orders.isEmpty())` before calling
   `threadFactory.createSampleThread(...)`. With real factories now returning populated lists,
   this guard is always true in production. It remains as a safety net for stub scenarios.
   Idempotency path (skipped-already-seeded) still short-circuits before factories run.

9. **Idempotency mechanism.**
   `DemoSeedRunner.run()` checks `clientRepository.count() >= SEED_THRESHOLD (6)` before any
   factory call. On second invocation with 6+ clients, returns immediately. The `seedIsIdempotent`
   test verifies this: counts before and after a second `runner.run()` call are identical.

10. **WSTEPNIE_PRZYJETE typo in switch expression.**
    First compile attempt had `WSTEPNIE_PRIJETE` (missing Z). Fixed inline — trivial typo.

### Items for Stage 2 Reviewer

1. **Duplicate-seed safety:** Idempotency relies solely on `clientRepository.count() >= 6`.
   If a real operator has created 6+ clients but no seed orders, the runner will skip seeding
   orders too. This is intentional (one guard for the whole run) but Stage 2 should confirm
   this is acceptable for the demo use case.

2. **WSTEPNIE_PRZYJETE backward transition:** `changeStatus` allows PRZYJETE → WSTEPNIE_PRZYJETE
   with no state-machine check. Stage 2 should verify this is intentional (free transitions per
   design principles) and confirm it doesn't break any downstream invariants.

3. **`picked_up_at` on WYDANE orders:** Set automatically by `OrderService.changeStatus` when
   targetStatus==WYDANE and `pickedUpAt==null`. Stage 2 should confirm this is the expected
   mechanism (not manually set in seed).

4. **Audit under non-HTTP context:** `AuditLogAspect` (8-7) reads `Span.current()` for trace_id.
   Seed runs outside an HTTP request so trace_id will be null. 8-7 handles this per carry-forward
   facts. Stage 2 should verify no audit-related exceptions surface during the seed run.

5. **Photo seed absence:** The spec says "2-3 sample photos" but the plan explicitly defers this.
   Stage 2 should confirm the deferral is acceptable for M8 demo-readiness.

6. **Polish diacritics in client names:** `Wiśniewska`, `Wójcik`, `Zieliński` — these are Java
   string literals in UTF-8 source. Stage 2 should confirm the source file encoding is handled
   correctly by the build (Maven default is UTF-8).
