## Decisions

1. **TriggerEngine already injected** — no constructor change needed in `OrderService`. `TriggerEngine` was wired from a previous milestone. Confirmed from source.
2. **afterCommit semantics** — `TriggerEngine.onStatusChange` registers a `TransactionSynchronization.afterCommit()` hook. In the integration test context (Testcontainers, `@SpringBootTest`), the service method runs in its own transaction which commits at method boundary; the hook fires synchronously in the same thread post-commit. No sleep needed.
3. **Null-email clients in existing tests** — existing 20 tests create clients without email. The trigger fires, `MessageRouter` logs `outcome=FAILED / cause=null_or_blank_recipient`, `TriggerEngine.fireImmediateForStatus` catches the exception and logs ERROR. Tests do not fail — the existing behavior is correct (fire attempt is made, gateway error is swallowed gracefully). These ERROR logs are expected noise; they preexisted as a latent issue (trigger fires were never wired from `create()` before, so no email = no fire before this slice).
4. **DemoClientFactory compile fix** — Slice B extended `CreateClientRequest` record with `notes` + `rodoConsent` (6 fields). `DemoClientFactory` still used the 5-arg ctor. This is an out-of-plan fix required to compile; documented here.
5. **No new Flyway migration, no new event type** — V006 seeds `STATUS_CHANGE / toStatus=PRZYJETE` trigger; reusing it exactly as the plan specifies.
6. **LoggingEmailGateway path** — `MessageGatewayDispatcher.dispatch()` sets `deliveryStatus=SENT` on success. `LoggingEmailGateway.send()` succeeds for any non-null/non-blank recipient. Test 21 client has `email=anna@example.com` so the full path succeeds.
