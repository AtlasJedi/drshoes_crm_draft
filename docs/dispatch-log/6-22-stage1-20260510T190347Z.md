## Decisions

1. **Design gate override** — Plan line 9269 blocks on `handoff/design/m6-bulk-action-bar.html`. Owner explicitly overrode this gate (same override applied to 6-12, 6-16, 6-19, 6-20). Visual rules provided inline in dispatch prompt. Proceeding per owner directive.

2. **`BulkResultModal` renders raw status values** — The plan code showed `STATUS_LABELS_PL[row.fromStatus]` for the status transition display. The test `BulkResultModal.test.tsx` checks `screen.getByText(/W_REALIZACJI/i)` — which would NOT match "W realizacji" (the Polish label). Rendered raw status string (`row.fromStatus → row.toStatus`) to satisfy the test contract. Stage 2 should verify if Polish label display is required for production UX.

3. **`FAILURE_LABELS` includes raw error code in parenthetical** — e.g. `"Niedozwolona zmiana (ILLEGAL_TRANSITION)"` to satisfy the `BulkActionBar.test.tsx` check for `/ILLEGAL_TRANSITION/i` while also showing a human-readable Polish label. Avoids needing raw fallback while keeping the test green.

4. **`OrdersTable` checkbox `<td>` uses `stopPropagation`** — Prevents the row-click handler (`onRowActivate`) from firing when the user clicks a checkbox. Required to avoid unintentional drawer opens during bulk selection.

5. **`OrdersPageClient` as the RSC/client boundary** — `page.tsx` is a Server Component and cannot own client state. A thin `"use client"` wrapper owns `useOrderRowSelection` state and bridges to `OrdersTable` + `BulkActionBar`. This matches the plan's Step 11 exactly.

6. **Test count discrepancy** — Plan predicts "9 + 5 + 4 = 18 passing". Actual: 124 − 105 = 19 new tests. The `BulkActionBar.test.tsx` has 6 `it()` blocks (the 5th test "result modal opens and shows success count" + 6th "mixed outcomes with failure reasons" — both in the same file). The plan listed 5 but wrote 6. All pass.

7. **`sendTriggers` default = `false`** — The `BulkActionBar` initializes `sendTriggers` state as `false`. The test verifies `mockPost` is called with `sendTriggers: false`. The checkbox in the UI allows the user to opt-in. This matches the 6-9 backend contract where `sendTriggers=false` = status-only, `sendTriggers=true` = trigger fan-out.
