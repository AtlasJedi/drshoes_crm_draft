# Task 0b-4-fixup — Remove phantom CSRF route + add CSRF enforcement tests

1. `/api/admin/auth/csrf` was in `PUBLIC_MATCHERS` with no controller and no design mandate.

## Blocker 1 resolution

Removed `/api/admin/auth/csrf` from `PUBLIC_MATCHERS`. No controller exists for this path
and neither `DECISIONS.md` nor the milestone plan mandates a separate CSRF endpoint.
Per DECISIONS.md §Auth, the CSRF token is delivered as a cookie (`XSRF-TOKEN`) on the
login response via the double-submit pattern — no dedicated GET endpoint is needed.

## Blocker 2 — new test methods

1. `admin_post_without_csrf_token_is_rejected_with_403_not_401`
   — POST `/api/admin/orders` without CSRF token/cookie → expects 403 (not 401).
   CsrfFilter runs before auth; if CSRF were disabled this would return 401, making the
   test a genuine falsification of "CSRF is enforced."

2. `csrf_token_repository_is_cookie_based_with_http_only_false`
   — Injects the `CsrfTokenRepository` bean, asserts it's a `CookieCsrfTokenRepository`,
   materialises a token via `generateToken` + `saveToken` on mock request/response, and
   asserts the resulting cookie is named `XSRF-TOKEN` and has `httpOnly=false`.
   Uses the exposed bean rather than lazy materialisation (Spring Security 6 defers cookie
   writes to state-changing requests, making a simple GET unreliable for this assertion).

3. `webhook_post_without_csrf_token_is_not_rejected_by_csrf_filter`
   — POST `/api/webhooks/test` without CSRF token → expects status != 403.
   Proves `ignoringRequestMatchers` is wired for the webhook path (no controller yet → 404,
   which is not 403, so the assertion passes while still being a falsification).

## Decisions / deviations from plan

- Exposed `CookieCsrfTokenRepository` as a `@Bean` (`CsrfTokenRepository`) to enable
  reliable cookie-config assertion in tests. This is a small API surface addition but
  avoids brittle Spring Security 6 lazy-materialisation timing issues.
- `getCookieName()` does not exist on `CookieCsrfTokenRepository` in Spring Security 6.4.x
  (setter only); used `saveToken` + `MockHttpServletResponse.getCookie("XSRF-TOKEN")` to
  verify the cookie name implicitly (non-null cookie = correct name).
