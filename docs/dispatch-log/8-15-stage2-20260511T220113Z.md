## Audit Results (12 Points)

1. **Credential leakage — PASS.** `route.ts` forwards only `content-type` to Jaeger.
   Strict allow-list (one header, hardcoded). `Cookie`, `Authorization`, and all other
   browser headers are dropped. Correct pattern.

2. **CORS on proxy — PASS.** No `Access-Control-Allow-*` headers added. Browser posts
   to same-origin `/api/otlp`. CORS is not needed and not present. Correct.

3. **`propagateTraceHeaderCorsUrls: [/.*/]` — FOLLOWUP.** Propagates `traceparent` to ALL
   outgoing browser fetches, including any hypothetical external domains. Current admin UI
   has no external fetch calls, so no data leaves the same-origin network path. Acceptable
   for demo scope. Recommend restricting to `/^\/(api|admin)/` or same-origin pattern in M9.

4. **No body size limit — ACCEPTABLE.** Stage 1 log correctly states Next.js App Router
   default is 4MB (not 1MB). OTLP batches are typically < 100KB. No action required.

5. **No auth on `/api/otlp` — ACCEPTABLE (documented).** Code comment explicitly states
   demo scope + Jaeger is docker-internal. Port 4318 IS host-exposed in `docker-compose.yml`
   (`"${JAEGER_OTLP_HTTP_PORT:-4318}:4318"`) for local dev convenience. Production Cloudflare
   Containers deployment should NOT expose Jaeger to internet. This is an infra concern for
   the Cloudflare deploy milestone, not a blocker here.

6. **Hydration ordering — PASS.** `<BrowserOtelInit />` renders `null` — no DOM output,
   no hydration mismatch risk. Placement before `<AdminSidebar>` in JSX is benign.

7. **`"use client"` chain — PASS.** Correct pattern: server layout imports
   `BrowserOtelInit` (client boundary in `BrowserOtelInit.tsx`) which in turn imports
   `browser-client.ts`. The `"use client"` in `browser-client.ts` itself is redundant
   but harmless — the effective client boundary is the wrapper component.
   Next.js will not attempt to execute `browser-client.ts` server-side because the
   import originates from a client component.

8. **SSR guard — PASS.** `if (typeof window === "undefined") return;` fires at the top
   of `initBrowserOtel()` before any OTel objects are constructed. Even if Next.js
   attempted SSR of this code, it would be a no-op.

9. **Tracer naming / service.name — MINOR FOLLOWUP.** Browser client defaults to
   `"drshoes-web"` (via `NEXT_PUBLIC_OTEL_SERVICE_NAME`), same as the server-side
   `instrumentation.ts`. Plan spec at line 2751 specifies `"drshoes-web"` — no separate
   browser name required by plan. In Jaeger, browser and server spans will show the same
   service name, making filtering harder. Recommend `NEXT_PUBLIC_OTEL_SERVICE_NAME`
   default to `"drshoes-web-browser"` in M9 for Jaeger UX clarity.

10. **W3C traceparent propagation — PASS.** `BasicTracerProvider.register()` with no
    `propagator` argument calls `_buildPropagatorFromEnv()`, which reads
    `OTEL_PROPAGATORS` defaulting to `['tracecontext', 'baggage']`. This registers
    `W3CTraceContextPropagator` + `W3CBaggagePropagator`. `FetchInstrumentation` with
    `propagateTraceHeaderCorsUrls` will inject `traceparent` headers on matching fetches.
    End-to-end W3C propagation is active. Verified in sdk-trace-base@1.30.1 source.

11. **sdk-trace-base version skew fix — PASS.** Importing `BatchSpanProcessor` from
    `sdk-trace-web` is correct. `sdk-trace-web@1.30.1` depends on `sdk-trace-base@1.30.1`.
    No separate `sdk-trace-base` in `package.json` — the transitive dep is consistent.
    No runtime version skew possible.

12. **Suite re-run — PASS.** 178/0/0/0 confirmed independently. Typecheck clean.
    Lint clean (0 warnings).
