## Decisions / deviations from plan

### 1. SmsApiProperties NOT injected directly — @Value used instead

**Plan implied:** The plan's carry-forward from 4-7 noted that `SmsApiProperties` already has
`callbackAllowlist` and `clientIpHeader` fields. The plan's controller template used `@Value`.

**Decision:** Followed `@Value` approach from the plan template. `SmsApiAutoConfiguration` is
`@ConditionalOnProperty(name = "messaging.sms.provider", havingValue = "smsapi")` — this condition
is false in both prod (provider=NOOP per application.yaml) and tests, so `SmsApiProperties` is
never a Spring bean. Direct injection would fail. `@Value` binds to the same property keys and
works regardless of whether the SMSAPI gateway is active.

`@Value` defaults match the 5 SMSAPI IPs from `SmsApiProperties` defaults, so prod behaviour
is identical to what `SmsApiProperties` would provide.

### 2. IT template bugs corrected

The plan's IT template (from plan line 3128) had four bugs matching the pattern from 4-7's
dispatch log note §7:
- `client.setFullName("SMS Klient")` → `setFirstName("SMS")` + `setLastName("Klient")` (Client entity)
- `order.setOrderType(OrderType.NAPRAWA)` → removed; `Order` has no `orderType` field
- `OrderType` import removed (class doesn't exist)
- `MessageEntity` instantiated via `MessageEntity.newMessage()` with `threadId`, `clientId`,
  `direction` all set (all NOT NULL constraints)
- Order `code` set to `"SMS-TEST-" + UUID...` (NOT NULL UNIQUE)
- `Channel.SMS`, `DeliveryStatus.SENT` replaced with string literals `"SMS"`, `"SENT"` (MessageEntity stores as String)
- `MessageRepository` from `com.drshoes.app.messaging.repository` (not `messaging.domain`)
- Added `@AfterEach tearDown()` with proper FK-order cleanup (matches PostmarkWebhookControllerIntegrationTest)
- Added `AuditLogRepository`, `MessageThreadRepository` injections for tearDown
