## Decisions

### LOC split applied
`ThreadController` (~100 LOC, GET only) + `ThreadMutationController` (~70 LOC, POST only). Both under 120-LOC cap per dispatch protocol. Reply endpoint (task 5-10) goes in a separate `ThreadReplyController`.

### LATERAL subquery for body search
The `searchThreads` native query uses a subquery inside EXISTS to match the latest 3 message bodies: `SELECT 1 FROM (SELECT body FROM message WHERE thread_id = t.id ORDER BY created_at DESC LIMIT 3) recent WHERE recent.body ILIKE ...`. This Postgres-compatible pattern avoids the `LATERAL` keyword while achieving the same correlated subquery semantics. Works correctly in Postgres 9.3+. The message-body search test is deferred (see Concerns).

### AdminPrincipal injection via authentication() not user()
The IT uses `SecurityMockMvcRequestPostProcessors.authentication(UsernamePasswordAuthenticationToken)` with an `AdminPrincipal` as the principal ‚Äî matching the pattern in `AdminWebTestBase.loginAsOwner()`. Using `.user("owner@drshoes.pl").roles("OWNER")` injects a `UserDetails` object which does not match the `AdminPrincipal` type parameter, causing `@AuthenticationPrincipal AdminPrincipal actor` to resolve to null. The IT uses a synthetic UUID `00000000-0000-0000-0000-000000000001` since no DB user is required.

### Audit FK warning is benign
The audit log aspect attempts to insert with `actor_id = 00000000-0000-0000-0000-000000000001` (the synthetic UUID). Since no real User row exists with that ID, the INSERT fails the FK constraint and the aspect catches it, logging `outcome=skipped`. This is the established pattern for tests that use synthetic principals without seeding a real DB user.

### Map.of() null key bug fixed
`Map.of()` (Java immutable map) throws NPE on `get(null)`. In `ThreadController.list()`, unmatched threads have `clientId == null`, so `clientNames.get(null)` was throwing. Fixed with explicit null-guard: `t.getClientId() == null ? null : clientNames.get(t.getClientId())`.

### @AfterEach teardown added
The IT seeds a `Client` ("Anna Kowalska") and threads, but `AbstractIntegrationTest` has no automatic cleanup. Without `@AfterEach`, the Kowalska client bled into `ClientRepositoryIntegrationTest` (which expects exactly one match). Added `@AfterEach tearDown()` cleaning in FK order: messages ‚Üí threads ‚Üí clients. The `@BeforeEach setUp()` also does the same cleanup for idempotency.

### csrf() added defensively to all POST tests (Fix-up D)
All 5 POST calls now include `.with(csrf())`. The CSRF filter is active on admin endpoints; without this, CSRF-protected endpoints return 403 not the expected status. The existing tests happened to pass without it due to test-class security config behavior, but adding it defensively is correct.

## Concerns / Follow-ups

1. **Message-body search test deferred.** The `list_searchByBody` test is NOT in the IT. The LATERAL subquery requires seeding messages with the correct `created_at` ordering and the HSQLDB test container (Postgres here, not HSQLDB) supports it, but reliable seeding of timed rows via `messageRepo.save()` is tricky since `created_at` is `insertable=false` (DB-managed). Deferred to a follow-up task or 5-21 hygiene.

2. **`lastMessagePreview` is null in all DTOs.** The `message_thread` entity/table has no `last_message_preview` column. The DTO field is always `null`. Frontend must handle this gracefully. If a preview is needed, a denormalized column should be added in a future migration (not in scope for M5).

3. **Synthetic actor UUID in IT skips audit log rows.** Tests using the synthetic `00000000-0000-0000-0000-000000000001` principal will never produce audit_log rows (FK constraint). If audit log assertions are needed for thread operations, the IT should be extended with `AdminWebTestBase` (which seeds a real DB user). Deferred to 5-21 hygiene.

4. **`searchThreads` channel=null handling.** The native query uses `:channel IS NULL OR t.channel = :channel`. Spring Data JPA passes Java `null` as SQL NULL. Verified this works with Postgres ‚Äî `NULL IS NULL` evaluates to TRUE. No issues expected in production.
