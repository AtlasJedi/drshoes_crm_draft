## Decisions / deviations from plan

### Status filter: SINGLE select, not multi-checkbox
Plan body said "multi-checkbox for status (6 statuses)". This is WRONG per task instructions
and backend reality: `GET /admin/orders` takes a single `?status=` param (confirmed in
`OrderController.java` line 68). Implemented as a `<select>` with "Wszystkie" empty option
+ all 7 statuses from `STATUS_ORDER`. Multi-status filtering is M1 debt.

### OrderListRow has no `items` array
`OrderListRow` (from 1-12) does not carry an items array — only `description`. The "Pozycje"
column renders `row.description ?? "—"` as a proxy. Proper item summary (kind labels joined
with " · ") requires either enriching `OrderListRow` on the backend or a separate fetch.
Documented here; not fixed (out of scope for 1-14).

### `OrderListRow` has no `assignedCraftsmanId`
The list row DTO does not include `assignedCraftsmanId`, so the "Wykonawca" column renders "—"
for all rows. The full `OrderDto` has this field; the drawer (1-16) will show it. Backend
DTO enrichment is M1 debt.

### Cookie-forwarding pattern
Mirrored `getMe` from `apps/web/lib/auth/session.ts` exactly:
- Uses `process.env["INTERNAL_API_BASE"] ?? "http://localhost:8080"`
- Calls `cookies()` from `next/headers`, builds `cookie:` header string
- `cache: "no-store"` on every fetch
- Created `lib/orders/api-server.ts` as the server-only module (not polluting the isomorphic `lib/api.ts`)

### Typed routes (Next.js 16 `typedRoutes: true`)
`router.replace()`/`router.push()` with template literals require `as Route` cast.
`Link href` for `/admin/orders/new` (route not yet built — 1-15) requires `as any` cast
since typed routes only accept known compiled routes. Added `eslint-disable` comment.

### lib/users added as thin new module
1-12 did not add `lib/users`. Added `types.ts` + `api.ts` (client-side) here. Server-side
`listUsersServer()` lives in `lib/orders/api-server.ts` alongside `listOrdersServer()` for
co-location; could be split in a later cleanup pass.
