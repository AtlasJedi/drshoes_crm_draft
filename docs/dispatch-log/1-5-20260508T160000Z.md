## Decisions / deviations from plan

1. **Signature: `next(int year)` with `next()` overload.** Plan test used `seq.next()` (no-arg),
   but the dispatch instructions mandated `next(int year)` for testability. Implemented both:
   `next(int year)` is the primary method; `next()` delegates to `next(Year.now().getValue())`.

2. **Integration test not unit test.** As instructed, test extends `AbstractIntegrationTest`
   (real Postgres + V001 PL/pgSQL function). Class named `OrderCodeSequenceIntegrationTest`
   rather than plan's `OrderCodeSequenceTest`.

3. **Year isolation via isolated years, not table truncation.** Used years 1999 and 1998 (never
   touched by any other test class). `@BeforeEach` deletes those rows from `order_code_counter`.
   This is safer than `TRUNCATE` in a shared container and avoids FK/cascade surprises.

4. **Concurrency test included.** 8 `CompletableFuture.runAsync` calls on year 1999 produce 8
   distinct codes — verified the PL/pgSQL `ON CONFLICT DO UPDATE` is atomic. All 8 pass.

5. **Implementation: JdbcTemplate, not EntityManager.** Plan body showed `EntityManager` + custom
   sequences, but the ERRATA/dispatch instructions explicitly required
   `JdbcTemplate.queryForObject("SELECT next_order_code(?)", String.class, year)`. Used JdbcTemplate.

6. **REQUIRES_NEW propagation.** Dispatch instructions recommended this; adopted so counter
   increments are committed even if the outer order-creation transaction rolls back, preventing
   re-use of the same code number.
