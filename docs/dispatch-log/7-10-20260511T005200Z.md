## Decisions / Surprises

1. **`toHaveClass("bg-green")` vs `bg-green/15` class token mismatch:** The plan's test uses `toHaveClass("bg-green")` but the plan's component code uses `bg-green/15`. In Tailwind, `bg-green/15` is a single class token — `toHaveClass("bg-green")` does not match it. Resolution: changed classes to `bg-green bg-opacity-15` and `bg-orange bg-opacity-15` so the `bg-green`/`bg-orange` tokens are present as discrete classes that `toHaveClass` can match. Functionally equivalent visual output.

2. **`_client` naming convention not suppressing ESLint warning:** The plan uses `{ client: _client }` destructuring as the "unused but intentional" signal. However, the project's ESLint config (Next.js typescript preset) does not set `argsIgnorePattern: "^_"` for `@typescript-eslint/no-unused-vars` — only the default `_` prefix does not automatically exempt named params here. Resolution: destructure `{ client }` normally and use `data-client-id={client.id}` on the button element. This uses the prop in live code (not just a comment), satisfies the linter, and is semantically correct — it anchors the button to the entity it will act on when 7-15 wires the modal.

3. **`ClientListTable.tsx` not updated:** The plan's task 7-10 file list does not include `ClientListTable.tsx`. The comment in that file says "replace RodoInline with shared import" but the task 7-10 commit in the plan (lines 2913–2916) only stages the 4 new files. The inline `RodoInline` is left as-is per the plan scope. Follow-up task or hygiene pass can swap the import.
