## Verdict: APPROVE-WITH-FIXUP

Two required fixups before proceeding to 4-5:

### Fixup 1 — pom.xml: replace `spring-boot-starter-web` with `spring-web`

**File:** `backend/libs/email-gateway/pom.xml`

**Issue:** `spring-boot-starter-web` in compile scope pulls `tomcat-embed-core:10.1.33` into
the microlib's compiled artifact — a library must never bundle a servlet container.
`RestClient` and `SimpleClientHttpRequestFactory` both live in `org.springframework:spring-web`.

**Change (replace the starter-web dependency block):**
```xml
<!-- REMOVE: -->
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-web</artifactId>
</dependency>

<!-- ADD: -->
<dependency>
  <groupId>org.springframework</groupId>
  <artifactId>spring-web</artifactId>
</dependency>
```

The version is managed by the parent `spring-boot-dependencies` BOM, so no explicit version needed.

### Fixup 2 — PostmarkAutoConfiguration: add `before=` ordering

**File:** `backend/libs/email-gateway/src/main/java/com/drshoes/lib/email/postmark/PostmarkAutoConfiguration.java`

**Issue:** With the current imports file ordering (`EmailGatewayAutoConfiguration` listed first),
when `messaging.email.provider=postmark` is set and `drshoes.email.provider=NOOP` (the hardcoded
default in `application.yaml`), both `LoggingEmailGateway` and `PostmarkEmailGateway` register as
`EmailGateway` beans → `NoUniqueBeanDefinitionException` at context startup.

**Change:**
```java
// BEFORE:
@AutoConfiguration
@ConditionalOnProperty(name = "messaging.email.provider", havingValue = "postmark")

// AFTER:
@AutoConfiguration(before = com.drshoes.lib.email.EmailGatewayAutoConfiguration.class)
@ConditionalOnProperty(name = "messaging.email.provider", havingValue = "postmark")
```

This ensures `PostmarkEmailGateway` is registered as the `EmailGateway` bean first, and when
`EmailGatewayAutoConfiguration` is processed, `@ConditionalOnMissingBean(EmailGateway.class)`
sees the existing bean and backs off.

## Strengths

1. **`isNetworkError()` design is correct and well-justified.** The helper correctly targets
   the `SimpleClientHttpRequestFactory` transport quirk without misfiring on HTTP-level exceptions.
   The `onStatus` suppression pattern ensures `HttpClientErrorException`/`HttpServerErrorException`
   never reach the retry catch path. The logic is clearly documented in both Javadoc and the
   dispatch log.

2. **IT coverage is thorough and the WireMock scenario pattern for retry testing is solid.**
   The use of WireMock scenarios (state machine: STARTED → attempt-2) to model retry behavior
   is idiomatic, and the `wm.verify(N, ...)` assertions provide exact call-count evidence that
   retry semantics are correct. The HTTP/1.1 factory fix for WireMock compatibility is properly
   isolated to the test `setUp()` — production autoconfig already uses `SimpleClientHttpRequestFactory`
   correctly.
