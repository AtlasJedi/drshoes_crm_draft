## Decisions / deviations from plan

1. **version=1 on creation, not 0.** JPA @Version increments on the initial INSERT (version column starts at 0 in DB, Hibernate increments to 1 on first flush). Plan smoke pseudocode said `expectedVersion: 0` — actual required `expectedVersion: 1`. This is correct JPA semantics; plan was approximate. Adjusted smoke accordingly.

2. **CSRF bootstrap required.** Unlike 0b-13 (login-endpoint smoke only), this smoke includes state-changing admin endpoints which require the XSRF-TOKEN double-submit cookie. Cookie is lazily issued on any response when it doesn't exist — a dummy POST with wrong token triggers a 403 that carries `Set-Cookie: XSRF-TOKEN=...`. Subsequent requests use that token. No code change needed; documented for future smoke authors.

3. **Actuator health, not /api/health.** Stack is healthy via `/actuator/health` → `{"status":"UP"}`. `/api/health` returns 404 (no custom health controller was implemented in M0A/M0B/M1). The docker compose healthcheck uses actuator. Smoke adapted.

4. **XSRF-TOKEN pattern.** curl's `-c` cookie jar captures only `Set-Cookie` headers that arrive; the XSRF-TOKEN is only set when the server generates a new one. Triggering the 403 route causes the token to be issued. Extracted via `grep XSRF-TOKEN /tmp/dr-cookies.txt | awk '{print $NF}'`.
