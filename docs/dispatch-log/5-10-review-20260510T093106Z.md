## Failure-mode analysis
- **MessageRouter.sendReply throws (gateway down):** `dispatcher.dispatch(...)` is the gateway call. If it throws an unchecked exception, the `@Transactional` rolls back the `saveAndFlush` (no orphan message row). Spring MVC will catch the unchecked exception and return 500. This is sensible — the caller gets a 500 with no partial state. ACCEPTABLE.
- **Null/blank body:** passes through all controller guards (body is not validated). `MessageEntity.setBody(null)` is set, then `dispatcher.dispatch(saved, recipient, subject, null)` is called. Behavior depends on the gateway implementation — likely results in a gateway error or empty-body send. A `@NotBlank @Valid` guard would close this.
- **Null channel in request:** `Channel.valueOf(null)` in `sendReply` would throw `NullPointerException`, which becomes a 500. The controller checks `!thread.getChannel().equalsIgnoreCase(req.channel())` — if `req.channel()` is null, `equalsIgnoreCase(null)` returns false, so the 400 channel-mismatch path fires. ACCEPTABLE for channel=null. But for channel as a completely missing JSON field (deserialized as null), the NPE path in MessageRouter is reachable if the controller somehow passed it through. In practice the equalsIgnoreCase check gates it — a null channel request hits 400 before reaching the router. Verified acceptable.
- **Channel mismatch check timing:** check happens BEFORE the `router.sendReply` call. Correct — no gateway call is made for invalid channel.
- **Unmatched and discarded checks:** both happen before channel-mismatch check. Order: 404 → 422 (unmatched) → 422 (discarded) → 400 (channel). This is a sensible cascade. Channel mismatch being last is slightly unusual (one might expect it second), but it does not affect correctness.
