## Decisions

### 1. ChangeStatusRequest default for `sendTriggers`

Plan suggested `@JsonProperty(defaultValue = "true")`. Used exactly that. Jackson respects the `defaultValue` attribute for `BeanDeserializer` when the field is absent in JSON — boolean defaults to `true`. Verified with existing `OrderController` single-order callers (no code change needed there, they just don't send `sendTriggers` in their request bodies and get `true` by default).

### 2. `@Size(max=100)` removed from DTO

Plan originally included `@Size(max=100)` on `orderIds` in the DTO. This would produce a 400 from `@Valid` binding before the controller's MAX_IDS guard fires. Since the plan requires 413 for >100 IDs, I removed `@Size` from the DTO and delegated the cap entirely to the controller. Documented in DTO Javadoc.

### 3. VERSION_CONFLICT bucket structurally impossible via bulk endpoint

The `BulkStatusController` reads `order.getVersion()` from the DB-fetched entity and passes it as `expectedVersion` to `ChangeStatusRequest`. This means the version is always current when `changeStatus` is called — VERSION_CONFLICT can only occur via a concurrent race between the read and write within a single bulk request (no lock held between the two calls). The test was adjusted:

- **Plan's test:** `WYDANE→W_REALIZACJI` expected to produce `ILLEGAL_TRANSITION` (because plan assumed a state machine guard). That guard does NOT exist (free transitions per M1 locked decision). WYDANE→W_REALIZACJI succeeds.
- **Plan's mixed test:** Expected `NOT_FOUND + ILLEGAL_TRANSITION`. Both assumptions are wrong.
- **Actual test:** Uses `NOT_FOUND` (missing UUID) + `NOT_FOUND` (soft-deleted order). The VERSION_CONFLICT and ILLEGAL_TRANSITION buckets exist in code for future use but cannot be exercised without either a race condition or a future state machine guard.

### 4. Audit semantics — 1 row per bulk request, not per-order

`AuditLogAspect` intercepts `BulkStatusController.bulkStatus` (controller pointcut) → 1 HTTP audit row per bulk call. `OrderService.changeStatus` has no `@Audited` annotation → 0 additional rows per order. Tests verify `auditAfter - auditBefore == 1` for all bulk calls regardless of how many orders succeed or fail.

### 5. `ILLEGAL_TRANSITION` bucket scaffolded for future use

The `IllegalStateException` catch in the controller maps to `ILLEGAL_TRANSITION`. No state machine guard exists today, so this bucket is unreachable in the current codebase. It is scaffolded so future status guards don't require a controller change.

### 6. BulkStatusController LOC

Controller is 98 LOC — within the <120 LOC granularity constraint.

### 7. No `@Audited` on BulkStatusController

Per plan comment and existing pattern: the AuditLogAspect HTTP pointcut already fires for the controller. Adding `@Audited` would double-audit. Controller has a Javadoc note explaining this.
